# 从成功的平台转型中学习

你必须能够基于清晰的输入识别策略，以确保平台转型的成功。回到使平台成功的基本要素，将帮助你实施成功的流程和技术。我们将评估的模型是建立能力以支持、设计、开发、部署和运营的原始金字塔，该金字塔确立了稳定性、安全性、适应性和可扩展性（*图 8.1**）。关于此模型的详细讨论发生在*第二章*。每一层和每一柱在开发成功平台中都发挥着关键作用。

图 8.1：DevSecOps 金字塔

当开始使用一个新的平台时，重点在于初始的开发、部署和运营，同时确保你仍然遵循稳定性、安全性、适应性和可扩展性的支柱。由于组织之间的需求不同，评估每个领域并分配详细的指标可能很困难。然而，分配一个初始指标可以帮助确定提高平台或转向不同提供商的实际障碍。

在多个组织中使用的标准指标是建立性能水平的指标。这些包括网络可用性、故障响应时间、用户查询时间和资源利用率等评估：

+   **服务级别协议**（**SLAs**）：这是客户和提供商之间的战略级讨论。

+   **服务级别目标**（**SLOs**）：这些是中级运营任务，当某些指标超出界限时通知组织。许多组织经常跳过这些中级连接点，尤其是小型公司。

+   **服务级别指标**（**SLIs**）：这是系统内任务的第一级工作通知。

一个例子 SLA 可能是 99%的可用性，SLO 是跟踪一定增量（如一周）内的常规可用性，而 SLI 将是当前状态指示器。每个都可能会出现在仪表板上。

设计这些元素需要了解平台成功所需的要素。虽然考虑每个元素为成功提供了绝佳的途径，但我们往往跳过了 SLOs。对于小型公司来说，制定中级目标可能很困难，即使在大型公司中，它们也仅仅成为 SLI 信息的汇总。这些汇总直接关联到 SLA，而不是建立独立的领域。相反，SLOs 并没有增加识别和修复问题的能力，对于许多人来说，它们成为了一个痛苦的思想练习。

下一节将扩展一些 SLA 格式，以加速平台的成功。

## 平台工程的服务级别协议

任何 SLA 的目标都应该是设定服务提供的整体目标。在我们设计和工程化的平台上，我们的主要目标如下：

+   **开发便捷性**：衡量代码提交的速度、新功能的发布或代码的替换率

+   **成功的测试**：评估测试覆盖率、成功的测试或与发布相关的工件

+   **推送到生产**：评估有多少预期功能被发布给客户

+   **实时支持元素**：评估故障后的恢复时间、升级时间和帮助台元素的反应时间

在建立 SLA 时，六个典型类别有助于确保一致性：

+   **定义**：确定客户的需求。例如，99%的可用性、每年少于 48 小时的停机时间、4 小时内响应支持电话、30 分钟内响应故障、多个管道可以同时运行，以及所有关键的安全漏洞在生产前都得到修复。

+   **文档**：创建一个定期报告 SLA 状态的交互。由于这些项目通常持续时间较长，因此每周、每月或每季度的报告作为仪表板活动的总结是最好的。

+   **同意**：对 SLA 过程来说，最重要的是客户和制造商之间就同意的 SLA 衡量正确的指标达成一致。

+   **监控**：任何定义的 SLA 都需要监控。监控本身是通过 SLIs 进行的，但 SLA 应该定义每个功能的监控方式。例如，对故障的响应可能是按故障进行的，或者随着时间的推移跨多个管道汇总。自动监控可以减少人工开销，提高平台的稳定性。

+   **测量**：SLA 应该定义到一个可接受的水平。虽然每年少于 48 小时的初始 SLA 可能是可接受的，但记录停机率增加了实用性。如果第一季度有 30 多个小时的停机时间，SLA 可能被不适当地扩展。

+   **报告**：SLA 应包括一个报告机制。如果存在一种补救这些缺陷的机制，那么一个会提醒客户潜在缺陷的 SLA 可能无效。这一类别与监控和文档步骤相关。这里最重要的是评估何时存在问题，客户需要通知，何时问题内部处理。例如，如果平均持续故障时间为 25 分钟，那么识别超过 30 分钟的异常以及确认平均值可能是有价值的。

在建立服务级别指标（SLIs）时，所有这些方面也都是有益的。每个 SLI 都是一个对 SLA 有贡献的测量指标。这里最重要的方面是创建自动化的系统来跟踪不同方面。如果你正在跟踪故障，SLI 将显示故障开始的时间、停止的时间、处理问题的责任人，以及可能显示解决挑战难度的层级。SLI 还会显示当前时间的可用性，可能汇总到整体 SLA 中。虽然 SLA 和 SLIs 提供了一个坚实的基础，但通过检查一些平台转型的成功案例，我们可以学到很多。

## 成功场景

几个商业平台已经通过快速生产成功交付了客户平台并提升了价值。我们将在本节中查看一些示例。

### GitLab

市场上最好的 DevOps 平台之一是 GitLab。该公司从 2011 年开发的初始开源模式扩展到 2014 年创建了一个商业模式，即 GitLab Inc.。公司使用订阅模式提供不同的产品级别，尽管所有产品都通过类似的 SLA 和支持相互连接。Gartner 将 GitLab 评为 2023 年 DevOps 平台类别的领导者。该公司完全远程办公，拥有来自 60 个国家和地区的 1,800 名团队成员，支持超过 100 万活跃用户和 3000 万注册用户。

导致 GitLab 成功转型的是对稳定性、安全性、适应性和可扩展性的重视。GitLab 从 GitHub 格式转型意味着许多用户已经熟悉了内置工具。将仓库使用作为基本管道框架的一部分强调了技能的延续。使用 GitLab 创建软件的基本方法与许多学生在高中和大学计算机科学课程中学到的相同。

这些功能在使用免费或购买的平台版本时与平台治理集成。机密信息可以嵌入并在项目内部共享，但不能在项目外部共享。个人访问账户塑造了访问权限，组织可以购买基于云的系统或本地服务器。同时，这允许共享管道、运行多个管道和进行数据分析。治理随后将安全自动化与供应链意识以及常规升级联系起来。支持多种安全扫描和工具。

在考虑适应性和可扩展性时，GitLab 通过使用先前构建的工具集提供了更快的 CI 管道构建。这些扩展增加了速度，并能构建对组织价值流有益的有效软件。GitLab 网站包含了许多公司购买其软件并达到新高度的使用案例：

+   **洛克希德·马丁（国防航空航天）**：管道构建速度提高了 80 多倍，数千个 Jenkins 服务器退役，系统维护时间减少了 90%

+   **CACI 国际（国防 IT 服务）**：安全扫描速度提高了 13 倍，劳动力和管理成本降低了超过 90%，并将超过 7 个工具整合到一个通用平台上

+   **NVIDIA（先进硬件）**：在第一年实现了超过 51% 的增长，并为整体平台提供了 99% 的正常运行时间

+   **Carfax（汽车零售）**：将工具链从 12 个工具减少到单个实例，专注于功能交付，在多年中创造了 20% 的部署提升，并在 SDLC 早期通过秘密检测以及容器和依赖项扫描识别了 30% 的漏洞

这些公司都经历了不同的成功。客户之间的共同点是，他们之前都没有使用过综合平台，与 GitLab 的集成创造了新的成功路径。平台是一个赋能工具，让你有更多时间构建和部署软件，而不是专注于平台建设。

关于 GitLab 平台的一个抱怨是，尽管代码合并和管道工作有效，包括生成式 AI GitHub Copilot 和 GitLab Duo 应用程序，但整体工作管理方面仍然不足。该平台已整合 GitLab Issues 来模仿其他工作管理系统，但缺乏一些操作方面。平台在构建和部署级别表现出色，但将这些功能应用于更大组织仍需要一些工作。一个很大的差距是通过 Scrum 和 Kanban 看板等工具可视化正在进行的工作，或为问题创建补充文档。尽管存在这些问题，GitLab 仍然是工具激发平台设计并使该平台作为完整的木工坊对他人开放的优秀例子，允许关注所需项目。

### Netflix

在模型另一边，一个成功将平台整合到日常实践中的知名公司是 Netflix。Netflix 是更广泛市场上成功容器和自动化系统使用的首批适配器之一。许多最初的无序工程实践，如**混沌猴**部署，起源于 Netflix。混沌猴及其相关工具，被称为**猴群**，测试了 Netflix 的操作环境，以查找尚未发生但最终可能发生的问题。

2020 年，Netflix 进行了重大工具回归，以将开发人员团队带入一个稳定的平台。布赖恩·利瑟恩领导了这些转型。他发现，随着公司规模的扩大，最初使用的微服务架构变得越来越碎片化。平台的目标是管理多个服务，实现发现，并允许上下文切换。一个造成问题的流程示例是使用 Bitbucket 来审查拉取请求，Spinnaker 来检查部署管道，以及 Jenkins 来分析构建失败，然后拥有多个用于操作状态指标的内部工具。每次部署都需要在工具之间手动推动，而不是使用一致的平台。

随着架构的持续增长，这些多次交互变得越来越复杂。不仅操作工具难以使用，而且种类繁多意味着开发者往往不知道添加到整体链中的新工具和流程。例如，一个团队在 Bitbucket 或 Spinnaker 上的捷径可能只出现在他们的本地文档中。此外，常见修复的解决方案并没有在整个平台上分发，而只出现在本地实现中。

Netflix 从构建平台架构并过渡到联邦平台**最小可行产品**（**MVP**）开始。一个关键方面不仅仅是将现有体验迁移到新平台，就像常见的**单点登录**（**SSO**）那样，而是确保完全集成，以便工具能够相互通信。这种通信使得从开发的角度来看，可以查看整个 SDLC，而不仅仅是其中的一小部分。一个创新的概念是能够使用集合，在平台中选择多个元素来构建定制的工作流程。然后，这些工作流程在所有级别上仍然可见。

从 Netflix 的经验中得出的一个关键教训是，平台必须做的不仅仅是整合业务功能。一个有效的平台必须整合工作流程并创建关于新功能的知识。在过渡期间，平台工程师必须能够展示新平台如何减少时间和精力。GitLab 的例子显示了多个减少时间和提供有用工具的实例。然而，快速浏览 Netflix 技术博客([www.netflixtechblog.com](http://www.netflixtechblog.com))可以发现，他们在平台实施过程中面临了许多挑战。总的来说，Netflix 继续取得成功，多个季度的毛利润率通常超过 35%。

成功的转换表明，已经整合平台工程的公司获得了明显的胜利。公司和组织总是更愿意讨论成功而不是失败。深入分析失败的转换更多地成为个人对不同实例的个人体验的练习，而不是已发布的用例。下一节将评估导致失败或不完整转换的一些方面。

# 评估失败的转换

建立服务级别协议和实际标准可以导致成功的转换，因为您将专注于稳定性、安全性、适应性和可扩展性。但同样的关注点也可能导致转换失败。对某一领域的过度关注可能会使平台在某些方面表现出色，但允许其他方面严重失衡。有时，失败的转换可能是一个意见问题，而不是实际应用问题。在推广平台用于内部或外部使用时，小事情如**用户体验**（**UX**）、工具集成和更新流程可能很重要。以下是一些挑战的例子：

+   **小挑战**：这些例子包括用户满意度、工具集成和更新流程。这些影响生产时间、用户体验和支持工单指标。

+   **大型挑战**：这些挑战的例子包括缺乏架构和愿景的协调一致、持续停留在进行中阶段而不是完成，没有可工作的平台、与用户群体的分离，以及没有简单的绿色到棕色场的变化。在这些情况下，有用的指标不可用，因为用户尚未采用平台。

认识到潜在挑战是至关重要的第一步，但真正理解这些挑战是平台工程师所需的一项技能。这些主题将在后续章节中详细讨论。

## 小挑战

这些小挑战出现在以下 Atlassian 的完整用例中。导致这种失败的原因始于基于开发者不满意的糟糕用户体验、大量支持工单以及生产时间增加。解决这些问题需要对现有平台进行流程和技术上的改变。团队通常会局部解决问题，但挑战在于沟通和将这些修复整合到更大的平台中。

用户可能很挑剔，现有平台中的微小变化可能会引起高度的不满。在不同区域之间移动标准元素需要通过如 alpha-beta 实施等模型进行评估。例如，改变存储库结构、提供新的工具选项或推广本地快捷方式。这需要允许一些用户测试 beta 版本并提供反馈，然后再进行更广泛的推广。这种 alpha-beta 模型可以防止由于没有为新培训和文档制定明确路径而改变常见路径导致的大规模更改阻碍用户的生产。这些示例方法可以帮助解决用户的不满，而另一个有效的指标是提交和解决的支持工单。

支持工单始终是平台体验的重要指标。缺乏全面的支撑团队是转型不成功的关键指标。DevOps 表示，构建软件的团队运行它，但从小团队扩展到大型用户群体可能会损害这些改进。有效的支持模型有三个级别，以帮助维持 DevOps 的改进：

+   **第一级**：有关平台使用的信息在文档中存在的问题

+   **第二级**：需要管理员协助的配置更改

+   **第三级**：错误或新功能

为了澄清，错误发生在根据原始交付应该工作但未工作的情况，而功能是指用户想要但最初范围不包括的东西。对于新平台，初始的第一级工单数量会很高，然后这些数量很可能随着用户对操作更加熟悉而转换为更多的第二级工单。第三级工单数量应该始终很低，因为错误通常出现在开发和测试中，而新功能建议有助于设计下一步。

最后一个常见的挑战发生在工具集成和更新过程中。当讨论 GitLab 时，我们看到了文件和代码如何轻松地与项目关联，转移到管道中，并达到生产状态。在较差的平台中，你可能会看到代码存储在不同的区域，而文档和工作进度由不同的工具管理，缺乏联系。例如，编码工作可能在一个工具中完成，而代码更改则报告在不同的管理工具中，而不是两个工具相互连接。这种差距意味着当开发者提交和合并新代码时，他们必须进入一个新的系统记录更改。改变系统可能会减慢生产速度并影响整个团队的可观察性。

工具集成之后是平台更新过程。持续使用 DevOps 标准更新软件需要不断的更新。更新管理应该对用户透明，或者发生在设定的时间。透明度包括在更新发生时通知用户，并提供高级信息和链接到更多详细信息。这些更新可以是功能更改，与新教程或安全更新相关，甚至可以沟通后端更改以改善性能。定期更新与用户建立联系，告知用户何时解决已记录的问题和基本修复。当发布新的安全补丁时，用户可能会向支持功能大量请求更新。提前应对这一需求可以创建一个有效的平台。尽管解决了小变化，但许多平台在更重大和广泛的问题上仍然存在挑战。

## 大型挑战

如前所述，大型挑战可能包括愿景和架构之间的不良集成，无法达到生产平台，缺乏用户响应，以及没有棕色地带到绿色地带模型。这些问题中的任何一个都可能导致生产中断，而它们的组合将把一个良好的平台愿景转变为持续的资源消耗。解决这些问题需要组织内部的协调，面对自上而下和自下而上的问题。一个常见的错误是 DevOps 中缺乏在验证和有效渠道中的沟通或反馈。这种分析回到了康威定律，即一个组织生产的系统往往反映了组织的结构，具有等级制的组织创造等级制的工具，而扁平化的组织则创造扁平化的模型。

无法有效地完成转型是常见的问题。这种情况最常发生在团队试图构建他们的平台时。转型愿景始于在早期章节中展示的有效规划转化为架构。团队往往有一个期望的愿景，但缺乏架构计划。一个例子是那些有全面的周交付计划和优秀的五年计划的团队，但缺乏将交付与整体战略连接起来的中间部分。一些工程师在没有架构集成的情况下，从大问题跳到立即的解决方案。结果是，在许多不同的领域都有优秀的代码，但要么没有全面的集成，要么需要许多手动捷径才能成功。

另一个关键因素可能是标准指标分析中的误导性指标。一个平台可能成功地使一个应用程序达到生产状态，但其他应用程序却失败了。虽然一个应用程序可能部署成功，但平台工程已经失败了。成功的平台通过提供具有意见但非规定性解决方案的集成基础，使多个应用程序能够成功。一些公司可能在某个平台上比在其他平台上更好地推动某些元素，从而在短期内取得成功，但长期会带来复杂问题。我们经常看到这种情况出现在特定的代码平台，如 Python 或 Java，或者特定的功能平台，如测试或安全。这些可能是短期解决方案，但不足以达到理想的平台最终目标。

转型过程中平台失败的一个早期迹象是用户反应。部署初始平台涉及工程构建、协调**基础设施即代码**（**IAC**）以及将产品推向生产。在这个过程中，你必须记住要衡量用户体验。用户倾向于喜欢使用某些工具和功能。如果你移除这些功能，即使是为了积极的工程结果，用户可能会决定不使用该平台。一个常见的例子是平台在未告知用户的情况下为用户执行操作。这可能导致用户重复代码，例如，这可能会破坏预期的功能。平台必须关注用户满意度，并在其功能上保持透明。如果我们回到第二章的木工店例子*第二章*，你可以提供预切割的木材长度并展示设计元素，但用户会希望看到所有平台选项。从技术角度来说，这可以简单到提供下拉菜单，或者一个批准按钮。

许多平台失败的另一个关键领域是未能提供足够的使用说明。这种失败最常见于将棕色地带解决方案转换为绿色地带解决方案。开发者之前构建了一个应用程序，然后希望迁移到新平台。在构建他们的应用程序时，开发者可能会做出不再可用的选择。好的平台必须提供轻松对齐这些模型的方法。这个领域的常见失败与编程语言、数据库模式或测试版本有关。例如，一个应用程序可能基于早期的安全测试被提升到生产环境。当平台包含最新的安全版本时，该应用程序不再通过管道。通知用户开发和生产如何影响他们以及相关的平台解决方案可以减轻这些挫折。向用户通知解决方案可以是一个 GenAI 解决方案表现优异的领域，自动执行该过程而不是需要手动解决方案。以下两个案例研究是我们看到的不成功的平台实施示例。

### Atlassian

Atlassian 以其为管理 DevOps 转型的公司提供最佳起点之一而闻名。Jira 和 Confluence 等工具在商业市场广泛使用。Jira 提供定制的看板或 Scrum 解决方案，该解决方案扩展到多级项目管理。一些 Jira Cloud 的新实例甚至提供对多个团队的高级管理。Confluence 提供与 Jira 解决方案完全交互的文档和笔记，并补充这些功能。然而，据我们观察，Atlassian 有一个失败的平台。

该公司提供与其他功能如 Bitbucket 管理代码和 SonarQube 解决安全问题的集成解决方案。虽然每个功能都显示在平台上，但它们缺乏适当的集成。Bitbucket 是 Atlassian 套件的一部分，但 SonarQube 是一个推荐的插件，需要额外的集成。用户将 Jira 工单和发布版拉入 Confluence，反之亦然，但机制往往很复杂。Jira 工单难以与 Bitbucket 结果或 SonarQube 报告连接。这种差异迫使开发者需要在多个窗口之间移动并复制粘贴数据以获得单一的真实来源。这些单一来源在其他平台上更容易获得。

同时，问题链接并不总是跨平台传达，这创造了一个具有挑战性的工作环境。可视化状态的变化可能需要启动一个新应用程序，而不是在当前窗口中观察。使用 Bitbucket 的实际代码，但需要在其他功能中记录更改，这对普通用户来说变得非常困难。平台问题始于所需的高级知识，并通过困难和过于复杂的行政工具继续下去。用户通常需要在每个工具中需要多个管理密码，而不是使用共享的管理账户来承担所有责任。

Jira 最常见的问题之一出现在工作流程过程中。Bitbucket 的管道元素不会自动镜像到 Jira，当 GitLab 集成到 Atlassian 工具中时，情况更加困难。此外，许多团队尽管所有项目都在一个共同的组织中，但仍然试图创建自己的工作流程。例如，一个团队使用待办、进行中和完成等阶段，而另一个团队则使用代码 1、部署、修订、代码 2、测试 1、部署 2 和完成。试图解决这些多个团队之间的进度变得具有挑战性。Bitbucket 链接可以复制到 Jira 票据中，但需要切换功能来确定进度。

Atlassian 在处理用户满意度方面的另一个简单失败是，尽管有多次用户请求，如他们的社区板所示，但他们从未在其平台上包含通配符搜索。通配符搜索是在数字字母字符串之前或其中使用特殊字符（如 *****）来查找项目。这个功能差距意味着搜索 Atlassian 需要知道所需对象的准确名称。没有通配符搜索，用户必须搜索多个组合或滚动通过文件系统来找到所需的对象。这个小缺失可能会给用户带来高度的不满。

Atlassian 存在的一个最终问题是其众多安全挑战。该公司多年来经历了多次失败，在过去 12 个月内发布了多个关键和高级漏洞。单个关键漏洞可能会使系统脱轨，但多个漏洞会影响系统性能和用户态度。每个漏洞都是不好的，但 2024 年的第一个关键漏洞在第一周就导致了来自 600 多个 IP 地址的 18,000 个端口的 39,000 次访问尝试。一些问题已经得到解决，但许多问题仍在继续或尚未出现。漏洞的数量本身表明系统中存在关键漏洞，未来可能被利用。

结合缺乏集成解决方案、工作流程挑战和用户满意度，突显了 Atlassian 工具中的关键失败。这些系统可能在一开始运行良好，但长期遵守关键平台创建支柱时，最终会失败。平台缺乏集成挑战了稳定性，由于多个漏洞，提出了安全问题，用户满意度下降，显示出缺乏适应性，以及最初可扩展的非集成工作流程在系统扩展时带来挑战。尽管在商业上取得成功，但如果从平台工程的角度来看，Atlassian 就是那种模式。

### 美国联邦政府

从个人角度来看，在我们与**美国**（**U.S.**）联邦项目和承包商合作的经验中，政府的软件开发工厂经常表现出不良的实践。许多问题并非源于政府工作人员或他们的交付能力，而是来自过时的行政要求。《联邦采购条例》（**FAR**），一份包含 37 章、2,000 多页的庞大文件，以及为各个机构提供的补充材料和成本核算标准，构成了这些挑战的大部分。尽管这些法规多年来一直在更新，但它们更多地倾向于较慢的材料和劳动力获取模式，而不是提供有效平台和随后的软件所需的快速标准。

许多政府机构人员变动率很高，无法维持进度。这些变动发生在工人和管理层。我们为一个项目工作，看到所有级别的员工流动率在 30%到 50%之间。这种流动给团队目标的实现带来了挑战。在另一个例子中，关键利益相关者坚持认为，由于能力最大化，保持季度计划不变是进步的关键，但在第一周就个人增加了 800 小时的必要任务。在其他情况下，人员流动问题扩展到领导层，他们带来新的计划和想法，彻底改变了以前的工作。

政府流程中的一个关键挑战，由 FAR 加剧，是关注劳动时间而不是完成情况。监管机构无法正确区分正在开发的软件与完成的交付，这使劳动时间比完成交付更加重要。例如，一个希望将遗留数据库应用程序迁移到平台的机构必须雇佣开发者一段时间，而不是根据可能的交付来签订合同。可视化这一点，意味着雇佣 8 名开发者，每周工作 40 小时，持续 12 个月，而不是允许公司在六周内以固定成本提出应用程序交付。考虑到标准开发者的时薪超过 100 美元，成本将是 1600 万美元，而可能的应用程序成本要小得多。

政府平台面临的另一个挑战是从棕色地带系统转换为绿色地带系统。美国政府系统如此庞大且根深蒂固，以至于它们不能在不破坏整个系统的情况下改变小部分。一个优秀的平台的可扩展性和适应性方面体现了持续改进，但某些遗留系统可能无法实现这一点。一个很好的例子来自 COVID 时代，当时一个州政府需要找到 COBOL 开发者来更新一个旧系统。这种转换之前已经需要，但数据传输和有限的合同机会被视为挑战。

同时，政府法规通常要求进行多次超出 DevOps 软件能力的人工测试。我们熟悉的一个组织被承包提供用户请求的季度更新。在完成前两次更新后，政府机构报告称他们无法进行测试，并要求开发者放慢更新交付的速度，尽管有明确的和合同规定的更新交付要求。所要求的放缓将导致每 6 到 9 个月才发布一个更新，这会导致用户不满，威胁到维护安全的能力，并影响软件性能。与 DORA 指标一样，平台的目标是在更新完成后支持持续交付，而不是当用户可用进行两周测试周期时。

这些例子主要关注美国政府的运营。当转向不同的模式，例如在中国使用的模式时，我们发现他们的大多数平台都是政府赞助的。政府赞助的平台是使用政府资金创建并在公开市场上销售的。在其他时候，中国强制使用特定的平台，限制了用户的选择，如下所示：

+   吉互取代了 GitLab。

+   微博取代了推特。

+   微信取代了 Facebook。

+   阿里巴巴取代了亚马逊。

中国共产党（CCP）逆向工程平台，忽视知识产权限制，然后授权一家中国生产企业。他们还禁止西方平台或任何类似的应用，只允许用户实施中国解决方案。随后，由于国家强制使用，指标被扭曲，超过 10 亿用户采用了该平台。这些国家强制措施只在威权政府地区而不是开放竞争中取得成功，因为除了安全性、可扩展性、适应性和稳定性之外的其他因素推动了使用。

许多政府平台缺陷是合同性的。其他是由于心态造成的。前几章强调了文化、领导和组织标准，并表明平台工程不仅仅是技术解决方案。人为地减缓平台交付主要影响安全和稳定性。然而，无法及时响应用户请求也挑战了平台的适应性和可扩展性。仅仅购买平台转型是不够的；根据康威定律，组织必须为变化做好准备。以下部分突出了一些可以推动成功转型和识别未来趋势的最佳实践。

# 平台成功的最佳实践

认识到一些加速交付、集成和随后增长的最佳实践是确保成功转型的重要关键。本节确定了最佳实践，制定了一些初步的指导方针，并提出了确保平台保持当前状态的方法。以下是一些最佳实践的快速列表：

+   服务平台。

+   认识到增长限制。

+   不要构建已经存在的东西。

+   建立指标。

支持平台的技术不断变化，我们必须确保存在一种合适的思维方式来预测这些变化。有多种方式可以整合变化；这些最佳实践作为指南而不是详尽无遗的列表。让我们更深入地探讨它们。

## 服务平台

我们永远不应该忘记，尽管技术起源，但人类构建、运营和使用平台。如果人类使用机器，它将需要某种形式的服务。许多组织忽视这一点或试图将支持责任转嫁给购买客户。平台内应提供某种形式的服务。服务示例包括嵌入式知识库、通用消息系统、常规通知和合格的服务人员。

知识库可以采取笔记页面和文档的形式。易于使用的工具可以将存储库中的代码注释或 Markdown 语言转换为清晰的文档。此外，系统可以提供更新、停机时间或建议培训的通知。常见的指标，如可用性和使用率，通常被隐藏而不是与用户共享。共享这些知识可以在多个层面上建立信心。

当这些初步支持尝试失败时，用户应该能够联系到人类或 AI 驱动的服务来解决他们的问题。支持人员的第一目标应该是将问题识别为一级，在文档中；二级，轻微的配置更改；或三级，一个错误或功能。所有问题都应在用户和管理员可观察的格式中进行跟踪，例如 ServiceNow 或另一个票务系统。这创造了问题可观察性，并允许快速对齐指标。AI 支持的一个关键例子是聊天机器人。这些提供了一个用户提问并获得定制响应的地方，引导他们通过具体的答案。

## 认识到增长限制

每个新的平台都希望成为最成功的，拥有最广泛的采用率和最高的利润。这些目标与平台提供支持的活动相冲突。成功的平台实施可能遵循 Netflix 作为用户平台而不是开发导向的模式，或者 GitLab，在那里服务负担落在用户身上。真正的 DevOps 平台工程涵盖了多个领域。最难的事情之一是意识到，虽然可扩展性很重要，但有时增长需要受到限制。

所有平台最终可能支持数百万用户，但增长是随着时间的推移发生的。你必须意识到现有平台中的性能限制。Netflix 从 2011 年的 2400 万订阅者增长到 2023 年的超过 2.2 亿；然而，大多数增长区域在任何一年都保持了 20%到 30%的稳定增长速度。任何平台模型都应该努力达到类似的增长速度，扩大规划能力和匹配客户增长的能力。客户增长是产生长期价值和利润的最佳模型。如收集用户反馈和市场情报的调查等研究方法可以帮助公司规划预期的增长水平。独角兽式的增长总是被希望，但很少实现，许多公司在短时间内无法处理 100%或更高的增长。

在我们以往的经验中，有一家公司决心通过提供一项技术特性来实现成功，为此投入了数千小时和美元来获得这种能力。然而，在他们完成开发之后，他们才意识到只有大约 10 个用户会使用这个特性，而且基于需求，用户数量不太可能增长。增长应该被仔细规划和讨论。小的变化和特性有助于增长，但需要与实际客户使用保持一致。这些一致性可以通过使用调查和运营指标来检查用户体验指标。

## 不要构建已经存在的东西

许多现有产品非常适合支持平台开发和部署。集成工具可以创造早期成功。许多这些工具要么是开源的，要么只需要最低限度的许可费用。在考虑工具选项时，第一步始终应该是工具是否可以通过开源或许可模式获得。这里的最佳实践是在考虑平台选项时，认识到何时可以租赁、购买和构建。

开源提供了一种解决方案，但并非唯一的解决方案。GitLab、Databricks、Elastic、MongoDB 和 JFrog 等公司以及其他公司，将它们的部分代码作为开源提供。如果通用解决方案是答案，这些可以在增长增加时早期采用，并切换到付费模式。提供商从开源切换到付费许可的能力使这成为租赁模型的一个例子。

供应商锁定是租赁模式的一个常见问题。这意味着如果你希望更改数据、架构或应用程序，它们不能轻易转换为新的模式。许多 DevOps 系统的互操作性减少了真正供应商锁定发生的频率。一个导致长期成功的好应用可能比在未来某个日期切换的能力更受组织青睐。这解决了敏捷和 DevOps 哲学，即团队必须能够轻松地改变，但这些变化也可以在紧密耦合的结构内发生。

购买软件可以是直接购买或通过订阅模式。订阅模式提供有限时间的解决方案，但选项过于受限时，可能会打开供应商锁定的大门。例如，Atlassian 提供订阅以构建平台能力，但如果你后来切换，则需要迁移数据库。有限的或过多的字段可能会在这里造成挑战。如果你打算长期保留软件，购买可以带来价值。最好的购买通常是安全、资源管理和项目管理。CI/CD、测试和其他部署流程经常变化，租赁模式在创造即时价值方面更有效。

最后一个选项是构建功能。这些构建选项通常是在你认为内部解决方案可以比市场上提供的更好时采取的最后一步。平台工程可能需要混合 IAC 和 **配置即代码**（**CAC**）构建，以推动其他租赁或购买的组件的集成。与其构建每个组件，现代平台架构允许在不进行密集准备的情况下将这些集成私有化。一个例子可能是购买一个存储库和 SSO 系统，你的组织创建一个集成，允许对存储库进行 SSO 控制。集成构建通常比全功能构建更轻量级。

## 建立度量指标

DevOps 总是需要度量指标。这些度量应根据你想要实现的目标进行定制。在构建过程中设置验收标准应决定组织随后要测量的指标。DORA 度量指标在初始开发实践中非常出色，因为它们提供了流速和容量测量，同时在后续测量修复和认可时间。你可以通过关于使用情况、可用时间、性能率、安全链接等方面的额外统计数据来补充这些指标。

包含度量指标的一个常见路径是之前提到的建立所需项目的 SLA 和 SLI 的路径。这些扩展到初始测量之外，在用户和供应商之间创建合同。将指标从测量转变为价值创造选项始终是最佳实践。我们之前在本章中详细讨论了 SLA。

记得用指标来衡量，建立往往会导致游戏化。在游戏化中，人们关注的只是实现指标，而不是背后的目标。你可以通过经常评估指标的成功或建立交叉参照指标来纠正这一点。一个很好的例子是在代码提交中：如果一个指标需要更频繁的提交，用户可能会缩小提交的大小；然而，达到整体目标仍然需要相同的时间。你可以将代码提交频率与客户时间和提交大小进行比较。这允许评估提交的最佳大小和时机，以实现客户价值。

# 创新以实现新的成功

虽然创新可以被认为是一种最佳实践，但它值得单独考虑。创新通常可以狭义地限制在下一个技术障碍上，但平台的所有方面都很重要。商业创新可能和推出新技术一样重要，因为它可以迅速带来更多客户。太多的公司倾向于只关注构建最新的技术，而不是改进内部实践。同时，落后于技术解决方案的商业模式可能无法支持价值或增长。团队应通过黑客马拉松、定期为特殊项目安排的停工时间、员工培训预算、参加会议、发表文章和广泛实验选项等做法来鼓励创新。

一个始终保持不变的平台支柱是稳定性。在考虑创新时，人们常常在区分**进化变化**和**革命性变化**之间挣扎。进化变化是从当前模型的小步骤，而革命性变化是重写底层模型。例如，BlackBerry 从手机上的数字键盘转向全键盘是一个进化变化，但 Apple 从手机转向今天的全功能智能手机是一个革命性变化。在采用这些变化之间找到平衡是商业的关键组成部分。历史上多层次的失败例子包括 IBM 声称个人电脑永远不会发生、柯达拒绝数字胶片、西尔斯拒绝从目录到电子商务的转变、Yahoo 在 1998 年以 100 万美元的价格没有购买 Google，以及 Atari 将游戏视为个人过程而不是共享过程。

重要的创新讨论之一是认识到**尖端技术**和**前沿技术**之间的区别。尖端技术指的是市场上可用的最新解决方案，而前沿技术通常是尚未广泛可用的技术。电视从管状到数字再到液晶到 LED，以及多代产品，提高像素数、色彩和传输能力的例子就是尖端技术。当前的前沿技术可能包括量子网络、神经连接设备或多功能人工智能助手。每种技术都解决了问题，但目前还没有市场或运营来支持更广泛的采用。

平台应尝试包括尖端解决方案，同时研究前沿技术，但不要期望客户会采用。尽管能力有所提高，但前沿解决方案往往会被诸如通用格式或可用性等问题所阻碍。一些例子包括激光唱片、Beta 和 VHS 格式的冲突，或者录音行业中的录音、8 轨、盒式磁带和 CD 之间的冲突。这两个领域随后都面临着数字媒体和流传输增长带来的挑战。

在创新和可用性之间找到正确的平衡，为理解如何深思熟虑地采用和整合新技术奠定了基础。

# 技术飞跃

理解技术变化是实现通过整合和适应实施变化的第一步。这引出了问题，我接下来应该考虑哪种技术？每个人都想找到下一个大事件，但革命性的变化很少发生。进化平台技术专注于几个组件：计算改进、存储增长和用户界面交互。这些每个都提供了好处，但应与整体战略相权衡。

## 计算

平台的第一个改进是提高计算能力。额外的计算通常需要更多的资金，并涉及在本地服务器或通过虚拟机添加额外的核心。从 CPU 核心转换为 GPU 核心提高了执行专用操作的能力。另一个可能因素是添加量子核心。虽然添加量子听起来很困难，但像 Amazon Braket 这样的能力使得无需购买这些解决方案即可使用量子机器。其他即将到来的变化包括以下内容：

+   **基于石墨烯的晶体管**：石墨烯使用单层碳原子厚度的材料，具有更高的连接性，卷成管状，占用更少的空间和能量。这些管状结构可以与其他二维材料结合，以增加传输速度。预计石墨烯纳米管将在未来五年内进入市场。

+   **神经形态**：涉及神经形态系统的技术需要复制人脑架构。在解决问题的差异将需要比传统模型显著更少的能量。这项技术落后，因为大多数尝试模仿生物过程的方法仍然涉及互联网协议，从而消除了改进结构的优势。

+   **光传输**：使用光方法进行传输使用光子而不是电子，通过光强度水平映射数据，并改变水平以执行计算。实现的光方法可以提高效率并降低功耗，但在集成和便携性方面仍存在挑战。

## 存储

在基础平台工程中，存储可能不像处理平台上的数据那样广泛需要。在某个时候，应用程序，尤其是 AI/ML 应用程序，需要更多的数据才能成功。使这些数据对平台可访问性创建挑战，特别是当产生数据扩展到千兆和太字节范围时。估计表明，全球每天的数据生产量超过 4 亿兆字节。大多数平台不需要这些水平，但做好准备意味着您可以向客户提供高级功能：

+   **DNA 数据存储**：将现有数据从二进制基础 2 转换为基于 DNA 的基础 4。这使得整体容量大大增加，以至于一些研究人员认为，全球的存储需求可以用一立方米的粉末状大肠杆菌 DNA 来满足。这种模式的挑战涉及使用蛋白质进行读写。

+   **钻石数据**：这涉及到使用激光脉冲在纳米级钻石中存储数据。这些技术具有读写能力，单个 5 厘米的钻石晶圆可以存储高达 10 亿张蓝光光盘，或约 25 艾字节。

+   **区块链**：区块链存储已经存在了一段时间，但仅限于特定的有限功能。将成功的区块链扩展到不同的和流式数据将提高系统的安全性和保密性。区块链存储的改进更有可能来自软件解决方案，而不是用户驱动的建议。

## 用户界面

通过技术改善平台的主要最后领域是用户界面。虽然用户界面可能看似边缘，但个人与平台互动方式的改进可以创造巨大的市场优势。一个很好的例子是 Netflix，从基于邮件的系统转变为按需流媒体改善了用户访问和查看数据的能力。这里突出了一些用户界面的顶级趋势：

+   **交互式 3D**：这种用户方法出现在许多科幻电影中，但其应用范围广泛。这将使用户能够查看、编辑并在 3D 空间中工作，而不是局限于传统的显示器和键盘配置。其实施意味着通过动作来访问和配置平台应用程序，例如用手势调用代码、运行管道和应用测试结果。

+   **虚拟和增强现实界面**：这些解决方案与交互式 3D 类似，但局限于特定元素。例如，3D 眼镜可以提供交互的外观，而不需要完整的 3D 显示。增强现实允许通过光学或音频输入提出额外的交互，可能在查看的应用程序中建议更改。

+   **虚拟助手**：如今，许多虚拟助手以聊天机器人和其他指南的形式存在。你甚至可以将语音激活的集成，如 Siri 或 Alexa，视为虚拟助手。对于平台来说，下一个改进可能将是 AI 跟踪日常活动以提出改进建议。例如，想象一个理解工人能力并能够根据他们在时间或质量方面的参数超出某些阈值时提供定制建议的 AI 系统，通知他们进行额外培训。

为未来做准备需要结合实施最佳实践和为新技术做准备。最佳实践包括人员和工作流程改进，如提供服务、创造增长机会和更新指标。技术变化需要认识到计算、存储和 UI 改进的下一步，例如量子计算、DNA 存储设备和虚拟聊天机器人。这些技术在先前的最佳和最坏案例研究中得到了展示，因为组织试图实现加速的平台工程。

# 摘要

从成功的转型开始，本章介绍了 SLA 框架作为 DevOps 成功的指南。引入 SLA 建立了用户与客户之间的指导方针，使成功或失败的判断成为可能。虽然不同平台之间的 SLA 可能非常相似，但成功的平台如 GitLab 和 Netflix 则采用了多条路径来实现整体成功。

相反，不成功的平台转型未能充分解决小规模和大规模平台建设挑战。小挑战往往与技术或运营相关，例如工具故障、服务质量差或不良指标。大挑战则更多是战略性的，平台无法使愿景与架构对齐，以及无法按时交付。DevOps 专注于持续的价值，因此任何影响交付的因素都可能导致失败。所强调的转型失败案例包括 Atlassian 和政府流程。

最后，本章通过确定一些高级最佳实践和平台创新即将到来的技术改进来结束。平台需要通过增加价值和扩大客户基础来成长。实现这一点需要有效地构建、整合现有技术、衡量成功并跟上创新。最佳实践提供衡量未来成功的指南。一旦确定了挑战，下一步需要确保存在测试和测量流程以保持持续的成功。下一章将探讨开发中的测试模型、交付中的质量保证和成功的运营平台。

行动呼吁

+   为平台制定由战略目标驱动的服务级别协议（SLA）和服务级别指标（SLI）。

+   常规评估该领域的其他人，以从他们的成功和失败中学习。

+   搜索新的技术进步，了解哪些是革命性和进化性的进步。
