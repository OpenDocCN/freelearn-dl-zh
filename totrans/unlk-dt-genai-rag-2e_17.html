Graph-Based RAG

Having built a robust ontology in Chapter 13, we now turn to implementing it as a functioning knowledge graph (KG). KGs have the potential to be the “brain” behind effective agentic RAG systems. Graph-based retrieval-augmented generation (RAG) involves querying a KG data source and presenting that data to the LLM for responding to queries. By using a KG, you can give your RAG results richer semantics and more explicit links between facts, enabling multi-hop reasoning that traditional retrieval methods cannot achieve.

In this chapter, we’ll take the financial ontology you created in Protégé and implement it in Neo4j, demonstrating how to build a graph-based RAG system that combines the formal reasoning of ontologies with the practical power of graph databases. In this chapter, we’ll cover the following topics:

  • Introduction to graph-based RAG
  • Code Lab 14.1 – building and querying a financial ontology KG in Neo4j

These topics will equip you with the practical skills to transform your ontology into a production-ready knowledge graph that can power sophisticated AI agents. Let’s begin by exploring what makes graph-based RAG such a powerful evolution beyond traditional retrieval methods.

Terminology: Graph-based RAG vs. GraphRAG

The name of this chapter is Graph-Based RAG, where we are intentionally avoiding calling it GraphRAG. “GraphRAG” is the name of a project that is related to what we will talk about here, but also has some key differences, and so we wanted to make sure we maintained the distinction between the two. Microsoft Research’s GraphRAG implementation is a full pipeline that uses LLMs to build, structure, and query KGs. It is a much more robust implementation of KGs than the simpler graph-based RAG we will outline here. It also has a specific way it groups entities and relationships, and a specific way that it conducts searches, which is different from the approach we outline in this chapter. It uses two modes of search: a global search that looks at high-level clusters of the graph to get an overview, and a local search that zooms in on specific related nodes for detail. At this point, it takes similar steps to what we outline in this chapter; the graph-based retrieval results are fed into the LLM to generate a final answer, and it has the same benefit of results that are structured and inherently relational, helping the LLM synthesize an answer that’s coherent and covers all necessary parts of the question. The experiments they ran show that GraphRAG could answer complex, multi-hop questions with greater accuracy than a standard RAG pipeline, helping to support how powerful KGs can be when used in RAG. Learn more about GraphRAG here: https://arxiv.org/abs/2404.16130.

The GitHub repo can be found here: https://github.com/microsoft/graphrag.

Introduction to graph-based RAG

When we talked about RAG in previous chapters, we relied on vector stores (semantic embeddings) and, at times, keyword (sparse) search to retrieve context for LLMs. While this hybrid semantic plus keyword approach improves relevance, it can still struggle whenever you need precise reasoning, explainability, or tight factual grounding. The next evolutionary leap is graph-based RAG, which leverages KGs to provide a structured, navigable context that dramatically enhances reliability, factuality, and multi-step inference.

The following are key advantages of combining KGs with RAG in AI agents:

  • Enhanced retrieval precision and relevance: By traversing explicit graph edges rather than relying solely on vector proximity, graph-based RAG filters out noise and returns highly targeted subgraphs that closely match the query intent.
  • Deeper semantic and contextual understanding: KGs encode entities, synonyms, and disambiguation paths, preserving nuanced meaning and preventing misinterpretation of polysemous terms. They can go far beyond what flat embeddings alone can capture.
  • Multi-hop reasoning and complex query handling: Agents can traverse relation chains (A→B→C…) to join disparate facts logically, supporting complex, multi-step inferences that traditional retrieval misses.
  • Explainability and traceability: Every retrieved node and edge comes with provenance metadata, enabling clear audit trails that are vital for regulated domains like healthcare and finance.
  • Holistic summarization and cohesive responses: Instead of disjointed snippets, graph-based RAG can extract and summarize connected subgraphs—yielding coherent overviews that respect the domain’s underlying structure.
  • Integration of heterogeneous data sources: KGs unify structured tables, ontologies, and unstructured text under one schema, letting agents query across silos without bespoke prompt engineering for each format.
  • Improved data freshness and updatability: Because graphs can be incrementally updatable, agents have the ability to always query the latest facts. If done properly, this can lead to cutting down hallucinations and eliminating the need for costly model retraining.

With those advantages in mind, let’s clarify the differences between how we’ve used KGs in previous chapters and how we use them in this chapter.

Cyclical graphs versus ontology-based KGs

In Chapter 12, we introduced cyclical graphs using LangGraph. These are structures where entities and relationships form feedback loops that enable recursive, context-dependent reasoning. They excel in domains with mutual influence, such as financial modeling, where a parent company’s decisions affect a subsidiary’s performance, which, in turn, shapes future strategy. Ontology-based KGs, which we’ll implement in this chapter, take a fundamentally different approach. They’re predominantly static, schema-driven structures built on directed acyclic graphs (DAGs), meaning you can’t follow edges and return to where you started.

At their core lies a taxonomy encoding “is-a” and “part-of” relationships, where the type hierarchy enforces acyclicity (no class becomes its own ancestor) while supporting clear inheritance patterns. For instance, task dependencies where Task A must complete before Task B, and Task B before Task C, form a DAG.

Figure 14.1 – Unidirectional information flow in DAGs

In contrast, cyclical graphs embrace loops. Consider a financial scenario where a parent company’s strategic decisions influence a subsidiary’s performance, which, in turn, shapes the parent’s future strategy.

Figure 14.2 – Cyclical graphs

These cycles emerge through conditional edges, mutual dependencies, or temporal feedback, enriching expressiveness but requiring cycle-detection and traversal strategies to prevent infinite loops.

This architectural difference matters for implementation. Cyclical graphs create active exploration loops where agents propose hypothesis-driven traversals and revisit nodes until reasoning converges. This is powerful for simulation but requires cycle-detection strategies. Ontology-based KGs enable straightforward lookups: the agent queries for definitions or constraints and incorporates them directly into prompts or post-processing. In RAG systems, they excel at filtering retrieval with precise type checks, ensuring that only valid, schema-compliant facts reach the LLM. Your financial ontology from Chapter 13, with its clear class hierarchies (Stock, Bond, Organization) and relationships (issuedBy, isRegulatedBy), is exactly this kind of stable backbone that grounds LLM responses in verified domain knowledge.

Core graph concepts refresher

A KG fundamentally relies on a clear, structured representation of entities and their relationships. The primary components of a graph are as follows:

  • Nodes (vertices): Nodes are entities or objects such as people, places, events, or concepts
  • Relationships (edges): Relationships describe the connections or interactions between nodes
  • Properties (attributes): Properties provide detailed context or attributes to nodes and relationships, such as names, descriptions, and timestamps

Social networks are often used to explain graph theory, where the people are the nodes, and the “connections” between them are the edges. These connections can be given values, such as a number representing how many times two people who are connected interact with each other. The implication is that people who interact more have a “stronger” bond, and this can be used in the analysis of the social network. But the most important aspect of graph theory, in my opinion, is that it isn’t just social networks. Anything can be represented by a graph. Any relationship between two things can be represented by an edge. As a result, KGs can really represent the hidden knowledge world that exists across pretty much everything out there that can’t just be captured in a traditional flat database. The more you work with graphs, the more you will realize how incredibly powerful they are in representing data, going way beyond rows and columns. This also helps to explain why they are so ideal to represent “knowledge” to your LLM in a RAG approach!

Important graph metrics include distance (shortest path between nodes), path length (number of hops in a connection), degree (number of connections a node has), neighborhood (adjacent nodes and relationships), and the relative “strength” of the edge/relationship properties. These metrics are crucial for efficiently translating user queries into meaningful graph retrieval operations.

Step 3.3 – parse the Turtle file into CSVs (new cell)

Neo4j doesn’t directly import RDF/Turtle files, so we need to transform our ontology into a format it understands: CSV files that clearly separate nodes (entities), edges (relationships), and properties (attributes). This intermediate step also gives us a chance to inspect and validate our data before loading it into the graph database, making debugging much easier if something goes wrong.

Run this cell to read your ontology and create ontology_nodes.csv and ontology_edges.csv:

g = rdflib.Graph()
g.parse('FinancialOntology.ttl', format='turtle')
# Helper to get first value of a given property
def get_first(subject, prop):
   for val in g.objects(subject, prop):
       return str(val)
   return None
# --- Collect nodes —
nodes = []
for s in g.subjects(RDF.type, OWL.Class):
    nodes.append({
        'id': str(s),
        'label': get_first(s, RDFS.label) or s.split('#')[-1],
        'comment': get_first(s, RDFS.comment),
        'type': 'Class'
    })
for s in g.subjects(RDF.type, OWL.NamedIndividual):
    nodes.append({
        'id': str(s),
        'label': get_first(s, RDFS.label) or s.split('#')[-1],
        'comment': get_first(s, RDFS.comment),
        'type': 'Individual'
    })
nodes_df = pd.DataFrame(nodes)
nodes_df.to_csv('ontology_nodes.csv', index=False)
# --- Collect edges —
edges = []
for s, p, o in g.triples((None, None, None)):
   if p in [RDF.type, RDFS.label, RDFS.comment]:
       continue
   if str(p).startswith('http://www.w3.org/2002/07/owl#'):
       continue
   if isinstance(o, rdflib.term.Identifier) and str(o).startswith('http'):
       edges.append({
           'source': str(s),
           'target': str(o),
           'type': p.split('#')[-1] if '#' in str(p) else str(p).split('/')[-1]
       })
edges_df = pd.DataFrame(edges)
edges_df.to_csv('ontology_edges.csv', index=False)
print("Created ontology_nodes.csv and ontology_edges.csv")
data_rows = []
for s, p, o in g.triples((None, None, None)):
    # Keep only literal values (data properties)
   if isinstance(o, rdflib.term.Literal):
       prop_name = p.split('#')[-1] if '#' in str(p) else str(p).rstrip('/').split('/')[-1]
        # capture datatype if present
       dtype = str(o.datatype) if o.datatype else None
       data_rows.append({
           'subject': str(s),
           'property': prop_name,
           'value': str(o),
           'datatype': dtype
       })
pd.DataFrame(data_rows).to_csv('ontology_data.csv', index=False)
print("Created ontology_data.csv")

Here’s an explanation of what we’re doing here:

  • Loads the exported Turtle ontology file into an in-memory RDF graph.
  • Defines a small helper to fetch the first value of a given RDF property, such as a label or comment.
  • Iterates over all ontology classes and individuals, capturing their IRIs, human-readable labels, optional comments, and type, then writes them to a nodes CSV:
    • Nodes CSV (ontology_nodes.csv): Every class and individual in the ontology gets a row with its ID (IRI), label (human-friendly name), optional comment, and whether it’s a Class or Individual type
  • Iterates over all object property triples (links between resources), skipping metadata and OWL schema terms, and writes them to an edges CSV:
    • Edges CSV (ontology_edges.csv): Every object property assertion (links between two nodes) becomes a row with source, target, and type (the property name)
  • Iterates over all data property triples (literal values attached to resources), recording the property name, value, and datatype, and writes them to a data properties CSV:
    • Data properties CSV (ontology_data.csv): Every data property assertion (a literal value assigned to a node, such as hasTicker = "AAPL") is captured with the subject node’s IRI, the property name, the literal value, and its datatype
  • These three CSVs together describe the nodes, relationships, and literal attributes your Neo4j import will use.

With our ontology successfully transformed into CSV format, we’re now ready to load this structured data into Neo4j and bring our KG to life.

Step 4.4 – load relationships from ontology_edges.csv

When we bring the relationships from your ontology into Neo4j, each one needs to be assigned a concrete relationship type (such as :ISSUED_BY or :OWNED_BY). In Cypher, you can’t just drop in a variable for the relationship type. It has to be written out directly in the query. Neo4j’s Awesome Procedures On Cypher (APOC) library has advanced tools for making that dynamic, but since we’re keeping this lab self-contained and simple, we’ll take a straightforward approach: we’ll only focus on the three edge types that matter for our financial ontology (issuedBy, isRegulatedBy, and ownedBy) and create a small Cypher query for each of them. Then, when we loop through the edges from the CSV, we’ll choose the right query depending on the edge’s name. This way, our ontology edges map cleanly into Neo4j’s relationship types without adding extra complexity:

edges_df = pd.read_csv("ontology_edges.csv")
print(edges_df.head())
# --- Define relationship type mapping ---
rel_map = {
    "issuedBy": "ISSUED_BY",
    "isRegulatedBy": "IS_REGULATED_BY",
    "ownedBy": "OWNED_BY",
}
# --- Cypher queries for each relationship type ---
query_issued = """
MATCH (a:Resource {id: $src}), (b:Resource {id: $tgt})
MERGE (a)-[:ISSUED_BY]->(b)
"""
query_regulated = """
MATCH (a:Resource {id: $src}), (b:Resource {id: $tgt})
MERGE (a)-[:IS_REGULATED_BY]->(b)
"""
query_owned = """
MATCH (a:Resource {id: $src}), (b:Resource {id: $tgt})
MERGE (a)-[:OWNED_BY]->(b)
"""
# --- Import edges into Neo4j ---
with driver.session() as session:
    count = 0
    for rec in edges_df.to_dict(orient="records"):
        t = str(rec.get("type", "")).strip()
        src = rec.get("source")
        tgt = rec.get("target")
        if t not in rel_map:
            continue # skip edges we aren't modeling here
    if t == "issuedBy":
        session.run(query_issued, {"src": src, "tgt": tgt})
    elif t == "isRegulatedBy":
        session.run(query_regulated, {"src": src, "tgt": tgt})
    elif t == "ownedBy":
        session.run(query_owned, {"src": src, "tgt": tgt})
    count += 1
print(f"Imported {count} relationships (ISSUED_BY, IS_REGULATED_BY, OWNED_BY).")

Here’s what the code does:

  • Reads the ontology_edges.csv file into a DataFrame and previews the first few rows
  • Defines a mapping from ontology property names to the uppercase relationship types to be created in Neo4j
  • Prepares separate Cypher queries for each relationship type (ISSUED_BY, IS_REGULATED_BY, and OWNED_BY)
  • Iterates through each edge record, filtering out any relationships not in the mapping
  • Runs the appropriate Cypher query to connect the source and target nodes with the correct relationship type
  • Counts the total number of relationships created and prints a summary message
  • At this point, it will list the relationships and output: 
    Imported 4 relationships (ISSUED_BY, IS_REGULATED_BY, OWNED_BY).

Step 6 – enable hybrid embeddings (text + structure) and multi-hop support

So far, your Neo4j graph is already powerful: it knows about classes (Stock, Bond, Organization, RegulatoryAuthority), individuals (AAPL, USTB, Apple Inc., SEC), their relationships (ISSUED_BY, IS_REGULATED_BY, OWNED_BY), and properties (such as hasTicker). This makes it excellent for precise queries with Cypher. For example, you could ask “Which stocks are issued by Apple?” and Neo4j would give you an exact answer.

But here’s the challenge: users often ask questions in natural language without knowing the schema, the Cypher query language, or the exact labels in your ontology. If someone types equities regulated by the SEC, your graph might not return a result directly because the node is labeled Stock, not Equity, and the user doesn’t know the IS_REGULATED_BY edge name. This is where hybrid embeddings come in, and this is why they are so powerful.

So, what are hybrid embeddings?

  • Text embeddings capture the language side of an entity (labels, descriptions, comments).
  • Graph embeddings capture the structural side (how nodes are connected, which relationships they have).
  • Hybrid embeddings combine both. They take the entity’s own information plus its connected context and flatten it into a natural-language string that represents the knowledge in that part of the graph. That string is then embedded into a vector space for semantic search.

This means each entity isn’t just represented by its name; it’s represented by a context-rich description that already encodes relationships and facts from the graph.

Here’s an example: AAPL. Instead of just embedding the AAPL label, we generate a hybrid description like this:

"Stock AAPL [AAPL] issued by Apple Inc. regulated by SEC — A security representing equity ownership in a corporation. Issuer regulated by SEC."

Notice what happened:

  • We combined the class (Stock), the label (AAPL), the ticker ([AAPL]), the issuer (Apple Inc.), the regulator (SEC), and even multi-hop information (the issuer itself is regulated by the SEC)
  • This creates a richer, semantically meaningful snapshot of AAPL that covers not only its direct properties but also its connected context

Why does it matter for RAG?

This technique is the secret sauce for making KGs dramatically more effective in RAG pipelines:

  • Improved recall: Queries such as equities regulated by SEC will find AAPL, even though the word “equity” doesn’t exist in the raw graph labels. The embedding captures the meaning.
  • Graph and vector synergy: Semantic search helps discover the right entities, and then Neo4j’s Cypher queries bring back the exact structured facts. It’s the best of both worlds.
  • Multi-hop reasoning baked in: By embedding enriched descriptions (up to N hops away), the system already knows second-order relationships before the search. That means when someone asks about an issuer’s regulator, the embedding has a good chance of surfacing the right entity even before Cypher expansion.

In short, hybrid embeddings transform your graph from a static database into a semantically aware retrieval engine. They bridge the gap between natural language queries and structured graph facts, giving your RAG pipeline the ability to understand both intent and ground answers in precise relationships.

This is one of the biggest reasons why KG-based RAG often outperforms traditional vector-only RAG.

Step 6.1 – build an enriched hybrid text (with multi-hop)

This step builds hybrid text descriptions for ontology entities, combining their own properties with related facts from connected nodes, so they can be embedded for semantic search:

def fetch_hybrid_text_for_class(class_label: str):
""" Returns a DataFrame with: id, class_label, hybridText for all :Resource nodes that are IS_A the given class_label, enriched with multi-hop info. """
# --- Cypher query with multi-hop traversal ---
    cypher = """
    MATCH (n:Resource)-[:IS_A]->(cls:Resource {rdfs_label: $class_label})
    OPTIONAL MATCH (n)-[:ISSUED_BY]->(org:Resource)
    OPTIONAL MATCH (n)-[:IS_REGULATED_BY]->(reg:Resource)
    OPTIONAL MATCH (org)-[:IS_REGULATED_BY]->(orgreg:Resource)
    WITH n, cls, org, reg, orgreg
    RETURN
        n.id AS id,
        cls.rdfs_label AS class_label,
        ( coalescecls.rdfs_label, '') + ' ' +
        coalesce(n.rdfs_label, '') +
        CASE WHEN n.hasTicker IS NOT NULL THEN ' [' + n.hasTicker + ']' ELSE '' END +
        CASE WHEN org.rdfs_label IS NOT NULL THEN ' issued by ' + org.rdfs_label ELSE '' END +
        CASE WHEN reg.rdfs_label IS NOT NULL THEN ' regulated by ' + reg.rdfs_label ELSE '' END +
CASE WHEN n.comment IS NOT NULL THEN ' — ' + n.comment ELSE '' END +
        CASE WHEN orgreg.rdfs_label IS NOT NULL THEN '. Issuer regulated by ' + orgreg.rdfs_label ELSE '' END
    ) AS hybridText
    ORDER BY n.rdfs_label
    """
    with driver.session() as session:
        rows = session.run(cypher, {"class_label": class_label}).data()
    return pd.DataFrame(rows)

Step 6.4 – embeddings and vectorstore

This step converts the hybrid text descriptions into numerical embeddings and stores them in a FAISS vector index so they can be quickly retrieved by semantic similarity:

class STEmbeddings(Embeddings):
    def init(self, model_name="sentence-transformers/all-MiniLM-L6-v2"):
        self.st = SentenceTransformer(model_name)
    def embed_documents(self, texts):
        return self.st.encode(texts, convert_to_numpy=True,
            normalize_embeddings=True).tolist()
    def embed_query(self, text):
        return self.st.encode([text], convert_to_numpy=True,
            normalize_embeddings=True)[0].tolist()
embedder = STEmbeddings()
# Build documents from your hybrid_df (from Step 6.1 / 6.2)

]

retriever = vectorstore.as_retriever(search_kwargs={"k": 5})

Here’s what the preceding code does:

  • Instantiates the embedder with the "all-MiniLM-L6-v2" model for lightweight, fast vector generation
  • Creates a list of LangChain Document objects from the hybrid_df DataFrame, storing each entity’s hybrid text and associated metadata (id and class_label)
  • Uses LangChain’s FAISS integration to build an in-memory vector store from these documents using the custom embedder
  • Creates a retriever from the vector store, configured to return the top five most similar documents for a given query

Step 6.5 – semantic search helper and graph expansion using LangChain FAISS

Use the LangChain FAISS vector store to search the hybrid text. The search_hybrid() function returns a DataFrame with id, class_label, hybridText, and score (score is the distance from FAISS; lower is better). We also add the expand_in_graph() function to give a list of Neo4j node IDs and fetch a small neighborhood (uses the driver from earlier steps):

def search_hybrid(query: str, top_k: int = 5) -> pd.DataFrame:
    results = vectorstore.similarity_search_with_score(query, k=top_k)
    rows = []
    for doc, score in results:
        rows.append({
                "id": doc.metadata.get("id"),
                "class_label": doc.metadata.get("class_label"),
                "hybridText": doc.page_content,
                "score": score
        })
    return pd.DataFrame(rows)
def expand_in_graph(ids, depth: int = 2):
    with driver.session() as session:
        data = session.run(f"""
            MATCH (n:Resource)
            WHERE n.id IN $ids
            OPTIONAL MATCH p=(n)-[*1..{depth}]-(m:Resource)
            WITH n, collect(distinct m) AS nbrs
            RETURN n.id AS id,
                n.rdfs_label AS label,
                n.hasTicker AS ticker,
                n.comment AS comment,
                [x IN nbrs | coalesce(x.rdfs_label, x.id)] AS neighbors
            ORDER BY label
        """, {"ids": ids}).data()
    return data

Here’s what the code does:

  • search_hybrid:
    • Runs a similarity search over the FAISS vector store to find the top k hybrid text entries most relevant to a query
    • Collects each match’s ID, class label, hybrid text, and similarity score into a list
    • Returns the matches as a pandas DataFrame for easier viewing and filtering
  • expand_in_graph:
    • Connects to Neo4j and retrieves nodes whose IDs are in the given list
    • Optionally matches neighboring nodes up to the specified number of hops (depth)
    • Returns each node’s ID, label, ticker, comment, and a list of its neighbors’ labels (or IDs if no label exists)
    • Outputs the results as a list of dictionaries for further processing or display

Step 7 – vector search and graph expansion (ready-to-prompt context)

We’ll wire the pieces together so that a natural language query is able to do the following:

  1. Gets embedded and matched against your hybrid text vectors.
  2. Pulls back the top entities.

Step 7.1 – helper: vector search and graph expansion

We conduct a semantic search over the LangChain FAISS vector store built in Step 6, and return a DataFrame with id, class_label, hybridText, and score columns:

        return str(s)

            "label": label})

                relationships["isRegulatedBy"][label] = clean(reg)
     # Format context with proper indentation

{rel_str}
}}

There is a lot going on in this code, so let’s try to break it down succinctly, starting with the vector_search function:

  • Performs a semantic similarity search in the FAISS vector store for the given query, returning the top k matches
  • Extracts each match’s ID, class label, hybrid text, and similarity score into a list
  • Returns the matches as a pandas DataFrame for easy viewing and further processing

This is similar to what we’ve seen many times now, setting up the code to do a vector search. We then add the graph_expand function:

  • Connects to Neo4j and retrieves all nodes whose IDs match the provided list
  • Expands each node’s neighborhood up to the specified number of hops (depth)
  • Returns each node’s ID, label, ticker, comment, and a list of its neighbors’ labels or IDs

This introduces newer concepts that are more specific to working with KGs, but fundamentally, we are conducting the data retrieval in this function.

The next function is probably the most interesting function, though: build_llm_context(). This function orchestrates the pipeline by searching, expanding, and formatting the results. But what is this format? This may seem different than anything we’ve seen in the past. This function converts our KG relationships output into Python static dictionaries. You have different options for how you want to present your KG data to your LLM:

  • Natural language: This is where you take the structured format from the KG output and turn it into natural language sentences. So, you might have text that says “AAPL is a Stock that is issued by Apple Inc. and is regulated by the SEC.” This approach is intuitive and easy to read, but can become verbose for complex multi-hop relationships and may introduce ambiguity that the LLM has to parse.
  • JSON: It is very common to see KG output in JSON format, where you have the nodes in one list and the edges in another list. The LLM naturally understands this format because it is so common and standardized, and it has seen a lot of the same kind of format in its training and fine-tuning.

However, we are introducing a third format – Python static dictionaries, based on recent research by Wu and Tsioutsiouliklis (2024) – which demonstrates that this representation significantly outperforms other formats for LLM reasoning. You will likely need to introduce dynamic elements to your data as well, which we do not do here, but in that case, you can add this right into this format because it is Python code already! See the information box for more information on this paper.

Thinking with KGs – enhancing LLM reasoning through structured data

Wu and Tsioutsiouliklis (2024) conducted comprehensive experiments comparing how different KG representations affect LLM reasoning performance. Their groundbreaking finding: programming language representations – specifically Python static dictionaries –dramatically outperform both natural language and JSON formats for multi-hop reasoning tasks.

The researchers tested three representation formats across multiple datasets. Natural language representation achieved modest improvements (44.6% accuracy), while JSON actually decreased performance (26.1% accuracy) compared to the baseline. However, the Python static dictionary format achieved 67.9% accuracy, which is a 78% improvement over the baseline. Even more impressive, when LLMs were fine-tuned with Python representations, accuracy jumped to 91.5%, surpassing much larger models. The study attributes this success to Python’s unambiguous structure and the fact that LLMs are extensively pretrained on code, allowing them to mentally “execute” dictionary lookups as explicit reasoning steps rather than ambiguous pattern matching.

This research fundamentally changes how we should approach KG integration with LLMs. By representing relationships as relationships['property']['entity'] rather than prose or JSON arrays, we transform vague semantic relationships into precise computational operations. The LLM doesn’t just read about relationships—it can traverse them like executing code. This is why our build_llm_context() function outputs Python dictionaries: it’s not just a formatting choice, it’s a scientifically-proven method for enhancing LLM reasoning accuracy by treating knowledge retrieval as program execution rather than text comprehension.

Reference: Wu, Z., and Tsioutsiouliklis, K. (2024). Thinking with Knowledge Graphs: Enhancing LLM Reasoning Through Structured Data. Available at https://arxiv.org/abs/2412.10654.

Step 8 – generate with LangChain and OpenAI

To complete our graph-based RAG implementation, let’s add the generation component that transforms our retrieved context into natural language answers. This final step demonstrates how all the pieces—ontology, KG, hybrid embeddings, and semantic search—come together to answer the competency questions we defined back in Chapter 13.

Now, let’s build a complete LangChain pipeline that combines our retriever with the prompt template and LLM to generate answers to user queries:

def make_context(question: str):
    bundle = build_llm_context(
        question, top_k=5, depth=2, max_neighbors=8)
    return {"context": bundle["context"], "question": question}
rag_chain = RunnableLambda(make_context) | chat_prompt | llm | 
    StrOutputParser()
user_queries = [
   "Is AAPL a stock or bond?",
   "What type of instrument is USTB?",
   "Which authority regulates MSFT?",
   "Which equities are regulated by the SEC, and who issues them?",
   "What stocks do you know?  What bonds?"
]
for i, q in enumerate(user_queries, start=1):
    print(f"\n=== Query {i}: {q}")
    try:
        ans = rag_chain.invoke(q)
        print(ans)
    except Exception as e:
        print(f"[error] {e}")

In this “pipeline” code, we see the following:

  • make_context: Calls build_llm_context to retrieve and format the knowledge graph context
  • RunnableLambda: Wraps the context function for use in the LangChain pipeline
  • chat_prompt: Applies the prompt template to guide the LLM’s response style
  • llm: Generates the answer using the structured context
  • StrOutputParser: Extracts the plain text response

The expected output is individual results for each of the five queries:

=== Query 1: Is AAPL a stock or bond?

=== Query 2: What type of instrument is USTB?

=== Query 4: Which equities are regulated by the SEC, and who issues them?
The equities regulated by the SEC are AAPL (issued by Apple Inc) and MSFT (issued by Microsoft Corp).
=== Query 5: What stocks do you know? What bonds?
I know about two stocks: AAPL and MSFT. For bonds, I know about USTB.
