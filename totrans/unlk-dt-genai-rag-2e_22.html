Advanced RAG with Complete Memory Integration

In Chapter 18, we built the foundation of procedural memory, which is a domain-agnostic learning system that enables agents to extract patterns from conversations and apply them at the appropriate hierarchical level. We saw how this system transforms static agents into adaptive ones, learning what works for different users, communities, and tasks. But that was just the beginning. Now, we need to integrate this procedural memory with the episodic and semantic memories from Chapter 17 to create a complete cognitive architecture, and explore the practical considerations that make these systems production-ready.

This chapter completes the journey by showing you how to create a fully integrated Cognitive Architectures for Language Agents (CoALA) agent that combines all memory types for truly intelligent behavior. We’ll explore LangMem’s different learning algorithms and when each excels, dive into the critical decisions around metric design that fundamentally shape how your agent evolves, and provide a complete framework for adapting this architecture to any domain. Whether you’re building a healthcare assistant, an educational tutor, or a customer service bot, this chapter equips you with everything needed to deploy production-ready learning agents.

Before we dive in, let’s clarify the relationship between the libraries and custom code in this chapter.

LangMem is a real, open source library from LangChain that provides the core primitives for memory extraction and prompt optimization (create_memory_manager, create_prompt_optimizer). However, LangMem itself doesn’t provide a complete agent architecture. Throughout Chapters 17 to 18, we’ve been building a custom framework on top of LangMem that implements the CoALA pattern with hierarchical learning, domain separation, and integrated memory types. The files you’ll work with, including coala_agent.py, procedural_memory.py, domain_agent.py, and the domain-specific implementations, are all custom code created for this book. They demonstrate how to architect production-ready learning agents using LangMem’s building blocks. All of this code is available in the GitHub repository, ready for you to use, modify, and adapt to your own domains.

In this chapter, we’ll cover the following key topics:

  • Code lab 19.1 – creating a full CoALA agent with integrated memory systems
  • LangMem’s learning algorithms: choosing the right approach
  • Designing domain metrics: the art of measuring success
  • Adapting to your domain: from investment advisor to any expert system

The foundation we laid in Chapter 18 with procedural memory now needs to be integrated with the episodic and semantic memories from Chapter 17. While Chapter 18 showed how procedural memory learns and adapts strategies, the real power emerges when all three memory types work together – episodic memory providing conversation context, semantic memory supplying extracted facts, and procedural memory applying learned strategies.

Let’s build this complete system and see how the memories reinforce each other to create truly adaptive, intelligent agents.

Code lab 19.1 – creating a complete CoALA agent with integrated memory systems

With the procedural memory system fully functional and tested, we’re ready to integrate it with the episodic and semantic memories from Chapter 17. Next, we’ll create the complete CoALA agent that combines all three memory types for truly intelligent, adaptive behavior.

Step 1 – create the complete agent with all memory types

Now, we integrate our domain-agnostic procedural memory system with the baseline agent to create a complete CoALA agent. This demonstrates how all three memory types work together through a clean, modular architecture where domain-specific logic remains completely isolated from the core memory systems.

We’ll start by importing the necessary components:

import os
import sys
import json
# Import the complete CoALA system
from coala_agent import CoALAAgent
from domain_investment.investment_advisor_agent import InvestmentAdvisorAgent

    process_baseline_conversations, test_agent_with_queries, 
    process_performance_feedback, process_remaining_conversations,
    test_hierarchical_retrieval, get_key_achievements
)
from dotenv import load_dotenv
load_dotenv(dotenv_path='env.txt')
os.environ['OPENAI_API_KEY'] = os.getenv('OPENAI_API_KEY')

The imports reveal the modular structure of our system. The CoALAAgent class is the domain-agnostic core that integrates all three memory types. The domain-specific components all come from the domain_investment directory, including InvestmentAdvisorAgent (which implements the domain interface), the data generator, and test utilities. This separation means switching domains requires only changing these imports to point to a different domain directory.

Next, we initialize our domain agent and create the full CoALA agent:

domain_agent = InvestmentAdvisorAgent()
domain_memory_dir = os.path.join(
    domain_agent.domain_dir, "domain_memory_store")
os.makedirs(domain_memory_dir, exist_ok=True)
# Create the full agent with all memory systems
full_agent = CoALAAgent(
    domain_agent=domain_agent,
    model_name="gpt-4.1-mini",
    temperature=0.0,
    persist_directory=domain_memory_dir,
    optimization_algorithm="prompt_memory"  # Can be "gradient" or "metaprompt"
)

InvestmentAdvisorAgent encapsulates all investment-specific logic: prompts, community definitions, success metrics, and task categorization. It also manages its own directory structure, keeping all domain data isolated. The CoALAAgent class receives this domain agent and automatically uses the domain’s memory directory for persistence, delegates all domain-specific decisions to the domain agent, and integrates all three memory types using the domain’s interface. The optimization_algorithm parameter selects which LangMem learning approach to use; we’ll explore the alternatives later in this chapter.

This test message exercises the full agent before any learning has occurred. The agent can respond using its base knowledge, but lacks the procedural strategies that would make its advice truly personalized. By capturing initial_stats before this interaction, we establish a baseline for measuring learning progress.

The key architectural achievement here is the complete separation of concerns:

  1. Domain agent creation: InvestmentAdvisorAgent encapsulates all investment-specific logic in the domain_investment directory. This includes the following:
    • Investment-specific prompts for learning and response generation
    • Community definitions (conservative retirees, aggressive millennials, etc.)
    • Success metrics specific to investment outcomes
    • Task categorization for investment queries

The domain agent also manages its own directory structure, keeping all domain-specific data and memory stores isolated. This means you could run multiple domain agents simultaneously without any interference.

  1. CoALA agent integration: The CoALAAgent class is completely domain-agnostic. It receives the domain agent and automatically does the following:
    • Uses the domain’s memory directory for persistence
    • Delegates to the domain for all domain-specific decisions
    • Integrates all three memory types using the domain’s interface

This clean separation means creating a new domain (such as healthcare or education) requires zero changes to the core CoALA agent; just implement a new domain agent following the same interface.

  1. Memory system integration: The full agent seamlessly integrates all three memory types from the CoALA framework:
    • Episodic memory: Stores complete conversations using the domain’s summary format
    • Semantic memory: Extracts facts using the domain’s extraction prompts
    • Procedural memory: Learns strategies using the domain’s learning prompts and success metrics

This integration enables what LangMem calls cross-reference analysis. When a user asks about rebalancing, the agent can do the following:

  • Recall their past conversations about investments (episodic)
  • Access their stored risk profile and goals (semantic)
  • Apply the most successful rebalancing strategy for similar users (procedural)

All while the domain agent handles the investment-specific interpretation of these memories.

Step 2 – load synthetic investment data

We’ll now load realistic investment advisor conversations that demonstrate various patterns, such as successful portfolio discussions, failed advisory attempts, and different client segments. This data will train our agent’s procedural memory to recognize what works, while maintaining complete isolation within the domain’s directory structure.

The data loading process is straightforward. We check for existing conversation data and load it, or generate new data if needed:

data_dir = domain_agent.data_dir
conversations_file = os.path.join(data_dir, "conversations.jsonl")
if os.path.exists(conversations_file):
    # Load existing data
    print(f" Loading existing conversation data from {data_dir}...")
    conversations = []
    with open(conversations_file, 'r') as f:
        for line in f:
            conversations.append(json.loads(line))
else:
    # Generate new data
    print(f" Generating new conversation data...")
    generator = EnhancedInvestmentAdvisorDataGenerator(seed=42)
    data = generator.export_realistic_data()
    conversations = data['conversations']
    
    # Convert to dict format
    conversations = [
        conv if isinstance(conv, dict) else conv.__dict__ 
        for conv in conversations
    ]

The data loading process demonstrates the domain’s self-contained nature. InvestmentAdvisorAgent manages its own data directory (domain_investment/investment_advisor_data/), keeping all domain-specific data isolated from the core system. This means multiple domains can coexist without interference; a healthcare domain would use domain_healthcare/healthcare_data/, an education domain would use domain_education/education_data/, and so on.

The synthetic data generator creates realistic investment advisor conversations with rich contextual information that LangMem’s data collection layer requires:

  • Conversation metadata:
    • Success indicators: Whether the conversation achieved its goal
    • Satisfaction scores: Client satisfaction ratings (1–5 scale)
    • Behavioral signals: Tracked patterns such as provided_specific_data and personalized_response
    • Query types: Performance analysis, rebalancing, risk assessment, tax planning, and so on
  • Data diversity: The generated conversations represent various investment scenarios and client types. This diversity is crucial for LangMem’s trajectory clustering capabilities; the system needs examples of both successful and failed interactions to identify what distinguishes them. The following applies to our investment advisor:
    • Some clients prefer detailed explanations, while others want quick answers
    • Certain approaches work better for retirees versus young professionals
    • Tax discussions require different handling than performance reviews

In other domains, this diversity would capture the following:

  • Healthcare: Different patient communication styles, symptom presentations, treatment responses
  • Education: Various learning speeds, subject difficulties, engagement patterns
  • Customer service: Issue types, escalation paths, resolution strategies

The behavioral signals in the data correspond to what LangMem calls “rich contextual information beyond simple logging.” These signals help the system identify subtle patterns – for instance, that successful technical troubleshooting includes diagnostic questions early in the conversation, while jumping to solutions correlates with failure.

The process_baseline_conversations processing function (imported from the domain’s test scenarios) encapsulates the complex learning pipeline while keeping the main code clean. This demonstrates how domain-specific processing logic can be modularized and reused across different experiments.

The following shows LangMem’s three-phase pipeline in action:

  • Extraction phase: The system pulls conversations into the appropriate memory stores. Complete interactions go into episodic memory for future reference, while specific facts get extracted into semantic memory using the domain’s extraction prompts. This comprehensive data collection captures what LangMem calls “rich contextual information,” and not just what was said, but how successful it was, what behavioral signals were present, and what outcomes resulted.
  • Analysis phase: The pattern mining process employs both explicit success indicators (conversations with success=true and satisfaction ≥4.0) and implicit behavioral signals to identify what works. This demonstrates LangMem’s success correlation analysis; only patterns from high-performing conversations become procedural rules. The system segments users into communities based on their characteristics, enabling what LangMem calls “trajectory clustering,” where similar user types are grouped to identify segment-specific patterns.
  • Synthesis phase: The learn_from_interaction method transforms identified patterns into concrete procedural rules at multiple scopes. Global patterns that work universally get stored at the global level, while user-specific preferences remain personalized. This hierarchical organization prevents overgeneralization, which is a critical safety feature that ensures advice appropriate for aggressive millennials doesn’t get applied to conservative retirees.

The community assignment logic demonstrates emergent segmentation. Rather than requiring predefined user categories, the system discovers meaningful groups through actual usage patterns. For our investment advisor, communities form around age and risk tolerance combinations. In healthcare, they might form around condition types and treatment responses. In education, they might form around learning styles and subject preferences.

The tracking and reporting provide the transparency that LangMem emphasizes. Every learned pattern can be traced back to the specific conversations that generated it, enabling developers to understand exactly why the system learned what it did. This auditability is crucial for production deployments where procedural learning must be explainable and reversible.

LangMem’s learning algorithms: choosing the right approach

While our code lab demonstrated procedural memory using the prompt_memory algorithm, LangMem provides three distinct approaches for extracting and optimizing behavioral patterns, each suited to different use cases and complexity levels. We will discuss these in the upcoming sections.

prompt_memory: efficient single-pass learning

The prompt_memory algorithm, which we used in Code lab 18-1, extracts procedural knowledge with just a single LLM call. It analyzes conversation trajectories to identify successful patterns and transforms them directly into actionable rules. This approach works well for most production use cases where you need quick adaptation with minimal computational overhead. The algorithm excels at identifying clear cause-and-effect relationships and is particularly effective for well-defined domains with relatively straightforward success metrics.

gradient: critique and proposal separation

The gradient algorithm takes a different approach by separating the critique phase from the proposal phase. It first analyzes what went wrong in failed interactions, building a comprehensive understanding of failure modes. Then, in a separate phase, it proposes specific improvements to address these issues. This critique-proposal split enables precise, targeted improvements and is particularly valuable when you need to understand not just what works, but why certain approaches fail. gradient is especially useful in high-stakes domains where understanding failure modes is as important as identifying success patterns.

metaprompt: multi-stage reflection for complex patterns

For more complex domains where patterns aren’t immediately obvious, the metaprompt algorithm employs multi-stage reflection. It first generates initial hypotheses about what might work, then iteratively refines these through multiple analysis passes. Each stage examines the patterns from different angles, capturing nuanced relationships that simpler approaches might miss. This makes metaprompt ideal for domains with subtle success indicators or where the relationship between actions and outcomes involves multiple intermediate steps.

Choosing the right algorithm

Your choice of algorithm should depend on your specific needs:

  • Use prompt_memory for production systems that need efficient, real-time learning with good-enough accuracy
  • Use metaprompt when dealing with complex domains where patterns require deep analysis to uncover
  • Use gradient when failure analysis is critical, and you need a detailed understanding of what doesn’t work

All three algorithms integrate seamlessly with the hierarchical learning structure we’ve demonstrated, automatically determining the appropriate scope for each learned pattern and maintaining the same safety features such as gradual rollouts and rollback capabilities.

Combining algorithms for comprehensive learning

While we’ve presented these algorithms individually, they can and often should be used together in sophisticated production systems. You might use prompt_memory for rapid initial learning during early deployment, then periodically run metaprompt analyses on accumulated data to discover deeper patterns that quick extraction missed. Meanwhile, gradient could continuously analyze failure cases in the background, ensuring that your system not only learns from successes but also systematically improves its failure modes. This multi-algorithm approach provides both immediate adaptability and long-term sophistication. For example, a healthcare system might use prompt_memory for real-time learning during patient interactions, metaprompt for weekly deep analysis of treatment patterns, and gradient to understand why certain interventions failed. The algorithms complement each other: prompt_memory provides quick wins, gradient prevents repeated failures, and metaprompt uncovers subtle optimizations that only become visible across many interactions.

How multi-algorithm learning works in practice

When using multiple algorithms together, they operate on the same conversation history but extract complementary insights at different depths and timescales. The system maintains all discovered rules in a unified procedural memory, using confidence scores and timestamps to resolve conflicts when multiple rules address the same situation.

The practical workflow follows a temporal hierarchy. During live operation, prompt_memory runs frequently on small batches of recent conversations, providing rapid adaptation to emerging patterns. These quick extractions catch obvious improvements, such as “users prefer specific numbers over vague statements,” within minutes of the pattern appearing. Meanwhile, gradient runs as a background process, focusing specifically on failed interactions, building a comprehensive understanding of what doesn’t work and why. This failure analysis prevents the system from repeating mistakes and identifies edge cases that success-focused learning might miss.

Periodically, perhaps daily or weekly, metaprompt performs a deep analysis of the accumulated conversation history. Because it examines hundreds or thousands of interactions together, metaprompt can identify subtle correlations that aren’t visible in smaller samples. It might discover that users who ask about volatility in morning hours require more emotional reassurance, or that certain phrase combinations predict follow-up questions with 85% accuracy.

These algorithms don’t replace each other’s findings; they build a layered understanding. Each rule is tagged with its source algorithm, confidence level, and the number of conversations it was derived from. When multiple rules could apply to a situation, the system prioritizes based on specificity (user-specific beats generic), confidence (rules validated across more conversations win), and recency (newer patterns may reflect changing user needs).

The accumulated rules create an increasingly sophisticated strategy library. A rule discovered by prompt_memory with 70% confidence might later be validated by metaprompt across thousands of conversations, boosting its confidence to 95%. Conversely, a pattern that seemed universal might be refined by gradient's failure analysis to include important exceptions. This continuous refinement happens automatically, with the system tracking which rules get applied, how often they succeed, and when they need updating.

The key insight is that different algorithms excel at different types of learning. prompt_memory provides immediate responsiveness, gradient ensures robustness by learning from failures, and metaprompt discovers the non-obvious patterns that separate good systems from great ones. Together, they create a learning system that adapts quickly, fails rarely, and continuously discovers deeper insights about what truly drives user satisfaction.

Designing domain metrics: the art of measuring success

The domain_metrics dictionary is where you encode what success means for your agent, and these choices fundamentally shape how your agent evolves. The metrics you choose and how you weigh them determine not just what your agent learns, but what kind of agent it becomes.

Single goal optimization: clarity with consequences

When you optimize for a single metric, the agent develops laser focus. An investment advisor optimizing solely for returns (100%) will learn aggressive strategies: leveraging margin trading, concentrating positions in high-growth sectors, and timing market volatility. The procedural memory will rapidly converge on patterns that maximize this single objective.

However, single-metric optimization often leads to unintended behaviors. That returns-only investment agent might learn to recommend risky options to conservative retirees because, historically, risk correlates with returns. A customer service agent optimizing only for speed might learn to close tickets quickly without actually solving problems. A healthcare agent focused solely on treatment adherence might become pushy and ignore patient concerns.

Multi-goal balancing: realism with complexity

Most production systems require balancing multiple objectives. Our investment advisor uses returns (50%), satisfaction (30%), and goal achievement (20%). This weighting creates an agent that primarily seeks performance but won’t sacrifice client relationships for marginal gains.

The priority weights act as “exchange rates” between goals. With our 50/30/20 split, the agent learns that it can accept a 3% return decrease if it increases satisfaction by 5% (since 0.03 × 50 = 1.5 points lost, while 0.05 × 30 = 1.5 points gained). This mathematical relationship shapes every learned strategy.

Consider different weightings for the same three metrics:

  • Conservative (20/60/20): Prioritizes satisfaction, learning strategies such as Always explain risks thoroughly and Confirm understanding at each step
  • Goal-focused (20/20/60): Develops strategies aligned with client objectives, such as Regularly review progress against retirement targets
  • Balanced (33/33/34): No clear priority, leading to moderate strategies that avoid extremes

Each weighting creates a distinctly different agent personality and behavioral pattern. Your choice of weights should reflect your true priorities, as the agent will optimize exactly what you tell it to – whether that aligns with your actual goals or not.

But what happens when your metrics inevitably conflict with each other? Understanding how the system resolves these tensions reveals much about how procedural learning actually works in practice.

Conflict resolution through weights

Conflicts between metrics are inevitable and revealing. When a high-return strategy consistently generates low satisfaction, the agent must choose based on their weights. This is where procedural memory becomes sophisticated; it learns conditional strategies, such as Use aggressive strategies only for users who explicitly prioritize returns.

The system handles conflicts by doing the following:

  • Learning segmented approaches: Different strategies for users who value different metrics
  • Developing conditional rules: For example, If user is risk-averse, prioritize satisfaction over returns
  • Finding creative solutions: Strategies that improve multiple metrics simultaneously
  • Accepting trade-offs: Explicitly choosing which metric to sacrifice when necessary

This conflict resolution capability is what makes multi-metric optimization powerful; rather than forcing a single approach on all users, the agent learns nuanced strategies that adapt to individual preferences and contexts. The procedural memory system naturally discovers these conditional patterns as it analyzes which approaches work for which users.

Beyond understanding how individual metrics interact, certain patterns in metric design have emerged as particularly effective across different domains. Let’s explore these proven approaches to structuring your metrics.

Metric design patterns

When designing your domain metrics, certain patterns have proven effective across different applications. Understanding these patterns helps you avoid common pitfalls and create metric frameworks that guide your agent toward genuinely useful behavior:

  • Complementary metrics: Choose metrics that naturally align. For healthcare, treatment effectiveness, patient satisfaction, and safety compliance work together as safe treatments tend to be effective and satisfying.
  • Tension metrics: Deliberately include competing goals to prevent extreme behaviors. For customer service, resolution speed versus customer satisfaction creates healthy tension that prevents both rushed closures and excessive time spent.
  • Guard rail metrics: Include metrics that act as safety constraints. For education, learning speed is the primary metric, but with comprehension and student well-being as guards against an overwhelming pace.
  • Hierarchical metrics: Structure metrics in tiers. Primary metrics (must exceed threshold), optimization metrics (maximize these), and monitoring metrics (track but don’t optimize). This prevents the agent from gaming the system by sacrificing critical requirements for marginal improvements.

Effective metric design typically combines several of these patterns. A well-designed framework might use complementary metrics for core objectives, tension metrics to prevent extremes, and guard rails to protect critical constraints, thereby creating a comprehensive system that guides your agent toward genuinely valuable behavior.

Understanding these patterns is essential for initial setup, but the real impact becomes clear over time. The next section explores evolution over time and how your metrics compound to fundamentally shape what your agent becomes.

Evolution over time

Your metric choices determine your agent’s long-term trajectory. An agent optimizing for user satisfaction develops increasingly empathetic, explanatory strategies. One optimizing for efficiency becomes increasingly streamlined and automated. Over hundreds of interactions, these differences compound into fundamentally different agents.

The procedural memory system will discover surprising optimizations within your metric framework. Given returns/satisfaction/goals weights, it might learn that discussing tax implications early improves all three metrics – a non-obvious pattern that emerges from the interaction between metrics.

Practical recommendations

Start with two or three core metrics that capture your essential objectives. More metrics create complexity without necessarily improving behavior. Weigh them to reflect your true priorities, remembering that equal weights rarely reflect real-world values.

Monitor for metric gaming – strategies that technically improve metrics while violating their spirit. An agent might learn to improve “satisfaction” by making unrealistic promises. Regular auditing of learned strategies helps catch these distortions.

Consider metric schedules. Starting with satisfaction-heavy weights during early deployment to ensure user acceptance, then gradually shifting toward performance metrics as the agent proves reliable. The procedural memory system adapts to these changing priorities, enabling controlled evolution of agent behavior.

Remember that domain_metrics isn’t just configuration – it’s the value system that shapes your agent’s personality, capabilities, and blind spots. Choose thoughtfully, as these metrics determine not just what your agent learns, but what kind of assistant it becomes.

Adapting to your domain: from an investment advisor to any expert system

The architecture we’ve built isn’t hiding its domain-agnostic nature; we deliberately isolated all investment-specific logic into the domain_investment directory. This clean separation makes it straightforward to create agents for any domain: healthcare, education, customer service, or beyond. Here’s the framework for adapting this system to your needs.

Domain conversion framework

To create a new domain (e.g., domain_educator for a tutoring agent), you need to convert four key files:

  • Domain agent (educator_agent.py):
    • Replace InvestmentAdvisorAgent with EducatorAgent
    • Update get_community_definitions(): Change from risk tolerance/age groups to learning styles/grade levels
    • Modify identify_task_type(): Swap financial tasks (rebalancing, tax planning) for educational ones (homework help, concept explanation, practice problems)
    • Adjust calculate_success_score(): Use weight comprehension (50%), engagement (30%), and progress (20%) instead of returns/satisfaction
    • Update domain_metrics: Track quiz_scores, time_to_mastery, and concept_connections instead of portfolio_performance
  • Domain prompts (educator_prompts.py):
    • Replace investment terminology with educational language throughout all prompts
    • Change portfolio to learning progress, returns to comprehension, and risk to difficulty level
    • Update the extraction focus from financial facts to educational ones (learning pace, subject struggles, and preferred examples)
    • Maintain the same JSON structure but with education-specific fields
  • Data generator (educator_data.py):
    • Replace portfolio templates with student profiles (visual learner, struggling reader, and advanced math)
    • Swap financial holdings for subjects and topics (algebra, biology, and essay writing)
    • Update conversation templates: How's my portfolio? becomes Can you explain photosynthesis?
    • Change success metrics from returns/satisfaction to comprehension/engagement scores
  • Test scenarios (educator_test_scenarios.py or integrated):
    • Convert investment scenarios to educational ones
    • Replace market volatility concerns with test anxiety scenarios
    • Update expected behaviors: explain jargon becomes break down complex concepts

These conversions maintain the same structural logic while translating domain concepts. The goal is to preserve the underlying patterns of user interaction and success measurement while adapting the vocabulary and context to your new domain.
