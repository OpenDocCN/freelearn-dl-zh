Procedural Memory for RAG with LangMem

Throughout our exploration of agentic memory in Chapter 16, we saw how the CoALA framework organizes cognition into four complementary memory types: working, episodic, semantic, and procedural. Working memory is basically chat history. In Chapter 17, we put episodic and semantic memory to work with hands-on labs. Now we arrive at the piece that turns a capable agent into an autonomous one. Procedural memory, the encoded know-how for performing tasks and the behavioral patterns that guide actions, marks the difference between an agent that merely remembers and one that learns, adapts, and improves on its own. In terms of the primary topic of this book, RAG, this chapter will show you how even the most advanced applications of generative AI still fall back on fundamental RAG principles. Without RAG, none of what we will discuss would be possible!

This chapter introduces LangMem, LangChain’s SDK for procedural memory optimization. With LangMem, an agent can review its own trajectories, extract stable behavioral patterns, and evolve its operational playbook through systematic prompt and policy updates. The result is a semi- or even fully autonomous agent that (1) learns for itself, (2) self-heals by detecting and correcting failure modes, (3) improves over time as every interaction feeds a learning loop, and (4) surfaces customer insights, including recurring intents, friction points, and resolution patterns that would otherwise take teams significant manual effort to uncover.

Here’s what procedural memory unlocks:

  • Autonomy: Delegate workflows end to end, not just single turns
  • Self-healing: Detect regressions, apply fixes, and promote better procedures
  • Compounding performance: Every interaction becomes training data
  • Customer intelligence: Mine patterns across chats to reveal needs, gaps, and opportunities without manual tagging marathons

In this chapter, we’ll cover the following:

  • Benefits of using procedural memory

The transformation from static to adaptive agents through procedural memory isn’t just a technical achievement but one that delivers concrete benefits across every aspect of agent deployment. From personalizing interactions at multiple hierarchical levels to dramatically reducing operational costs, procedural memory fundamentally changes what’s possible with RAG-augmented systems. Let’s examine how these capabilities translate into real-world advantages for your applications.

Benefits of using procedural memory

We’ve come a long way, progressing from basic RAG to its most advanced application: autonomous agents that can use retrieval as their data plane and procedural memory as the engine for not just improvement, but full recognition of user patterns that can automatically accelerate your effort to fulfill whatever goals you have as an organization and for your users. In other words, whatever you are trying to achieve as an organization, whatever your goals are, you can use this agent to help accelerate this effort in shockingly effective ways.

The applications of this approach are far-reaching. Once your agent reaches this level of autonomy, it can scale far faster than a purely human-in-the-loop system. It becomes a proprietary advantage over competitors (or a disadvantage if they get there first), because your agent will develop strategies to serve users more quickly and effectively than an entire team of marketers, product managers, and engineers. Most importantly, this capability is not magic; it is the natural outcome of building and using RAG in its most advanced form, where retrieval powers understanding and procedural memory transforms that understanding into continuous behavioral refinement.

Procedural memory enables agents to tailor their behavior at precisely the right level, whether for individual users, teams, tasks/intents, or entire organizations, through hierarchical namespace organization. The system captures rich contextual signals, including success indicators, satisfaction scores, and behavioral patterns, then segments these into meaningful groups to discover what works for each group. This deep understanding reveals not just what users ask for, but what actually drives their satisfaction and success. This segmentation approach naturally accommodates both common patterns and unusual scenarios. By maintaining separate learning tracks for edge cases while preserving common flows, agents can handle exceptional situations without degrading typical user experiences. The result is an agent that truly knows its users: their preferences, their pain points, and most importantly, the specific approaches that work best for each segment.

Through continuous analysis of conversation trajectories, procedural memory learns to produce more accurate, relevant responses while reducing hallucinations and dead ends. The system’s retrieval reformulation capabilities optimize query patterns and similarity thresholds based on what actually returns useful results, while failure-aware adjustments preemptively shift strategies when standard approaches are likely to fail. Templates for clarification, sequencing, and escalation keep conversations structured and productive. LangMem provides several algorithms for extracting and optimizing these patterns, with different approaches suited to different complexity levels. For this chapter’s implementation, we’ll use the prompt_memory algorithm, which provides efficient learning with minimal computational overhead. The comprehensive safety features, including A/B testing, gradual rollouts, and instant rollback, ensure that improvements are validated before full deployment, preventing regressions from reaching users.

Procedural memory automates the traditionally manual process of prompt maintenance and optimization across entire agent fleets. Pattern-to-rule transformation eliminates the need for developers to hand-craft prompt updates, while storage-agnostic APIs and LangGraph-native integration ensure seamless deployment at scale. The system accelerates onboarding through rule templates for common patterns, enabling new agents to immediately benefit from organizational learning. Continuous improvement happens without expensive retraining cycles as the core feedback loop automatically triggers optimizations based on conversation outcomes. This automation extends beyond simple logging to sophisticated trajectory clustering, cross-reference analysis between memory types, and temporal pattern detection that reveals how effectiveness changes over time.

By learning optimal retrieval strategies and conversation flows, procedural memory dramatically reduces both computational costs and time-to-resolution. The learning algorithms can optimize behavior efficiently, while learned diagnostic patterns eliminate wasteful regenerations and re-asks. Tuned retrieval thresholds prevent unnecessary document processing, and rule-driven flows streamline common interactions. The system’s significance testing and temporal stability checks ensure that optimization efforts focus on changes that deliver measurable impact. Perhaps most importantly, the reduction in maintenance overhead through automated updates, version control, and vendor-agnostic storage frees engineering resources for higher-value work while ensuring that every interaction makes the system more efficient.

These benefits aren’t theoretical but emerge naturally when you implement procedural memory in your own systems. Code lab 18.1 – implementing procedural memory optimization

In this lab, you’ll build a procedural memory system that enables any agent to learn and adapt its behavior at multiple hierarchical levels for a specific domain, but with the flexibility to apply it to any domain. This lab starts with the memory-enabled agent from Chapter 17, but we’ve consolidated it for you into the separate baseline_agent.py for convenience. The core of this chapter’s functionality lives in procedural_memory.py, a pre-built module that implements hierarchical learning, retrieval, and adaptation. Rather than constructing this system line by line, this lab takes a guided demonstration approach: you’ll run the code, observe how procedural memory operates at each scope level, and understand the architectural decisions that make it work. This approach lets you focus on grasping the concepts and seeing the system in action, rather than typing out infrastructure code. We’ll demonstrate how to create a modular system where the core learning mechanisms are completely separated from domain-specific logic, allowing you to easily create agents for any domain, such as investment advisory, healthcare, education, or customer service, simply by implementing a domain interface. We focus on investment advisors in this code lab, but at the end, we talk about how to convert this to your domain, whatever it is.

This implementation showcases two key architectural insights:

  • First, separation of concerns: The procedural memory system itself is domain-agnostic, working with any domain that implements the DomainAgent interface. All domain-specific logic prompts, community definitions, and success metrics live in isolated domain directories (such as domain_investment/ or domain_educator/), making it trivial to add new domains without modifying the core system.
  • Second, hierarchical learning: Not all patterns should be learned globally. The system automatically determines the appropriate scope for each learned pattern:
    • User-level: Capture individual preferences and personalized approaches
    • Community-level (group): Learn patterns that work for user segments sharing common characteristics
    • Task-level (intents, groups of actions, similar grouped concepts): Develop specialized approaches for specific types of requests
    • Global-level: Extract universal best practices that apply across all users

The procedural memory system allows agents to learn from their own successes and failures. After each conversation, the agent extracts what worked (or didn’t work) and stores these insights as concrete strategies. These learned strategies are stored as readable instructions that the agent can retrieve and follow in similar future situations. You can inspect, edit, or remove any learned strategy, giving you full control over how the agent evolves. Each strategy maintains its own success metrics and can be independently evaluated, updated, or rolled back.

This modular approach enables rapid domain-specific deployment. Here are some potential examples:

  • Healthcare: In medical settings, the procedural memory system learns from patient interactions to discover optimal care patterns. It might identify that elderly patients have better outcomes with morning appointments, that certain explanation styles improve medication adherence for diabetic patients, or that specific follow-up sequences reduce readmission rates. The system automatically segments patients by relevant characteristics (age groups, conditions, treatment responses) and learns what works best for each segment while maintaining safety compliance standards.
  • Customer service: For support operations, the system analyzes ticket resolutions to uncover patterns that improve customer satisfaction and reduce escalations. It learns that offering refunds early in certain situations prevents escalations, that enterprise customers respond better to technical deep-dives while individual users prefer step-by-step guidance, or that specific troubleshooting sequences resolve issues faster. The procedural memory automatically categorizes tickets and customer segments to apply the most effective resolution strategies.
  • Education: In educational contexts, the system observes tutoring sessions to identify which teaching techniques work for different learners. It discovers that visual learners benefit from diagram-based explanations, that certain pacing strategies improve comprehension for struggling students, or that specific practice sequences enhance retention. The system naturally groups students by learning styles and academic needs, continuously refining its approach based on comprehension metrics and learning outcomes.

The agent will learn what works globally, individually, task-wise, or group-wise, all while using the exact same procedural memory system that could power a healthcare or education agent.

Step 2 – define procedure structure for hierarchical learning

We’ll create a domain-agnostic data structure that captures learned procedures at multiple scopes, embodying LangMem’s comprehensive data collection capabilities. This structure will store success indicators, behavioral patterns, and adaptation history that enable sophisticated pattern analysis beyond surface-level metrics, while remaining flexible enough to work with any domain.

class DomainProcedure(BaseModel):
    """Generic procedure structure for any domain.
    
    Core fields (required for all domains):
    - strategy_pattern: Description of the strategy
    - steps: Ordered list of steps to execute
    - segments: Applicable user segments
    - success_rate: Historical success rate (0.0-1.0)
    - scope: Hierarchy level (global/user/community/task)
    
    Domain-specific data goes in:
    - domain_metrics: Dict for any domain-specific metrics
      (e.g., avg_portfolio_performance for investments,
             quiz_avg for tutoring, etc.)
    """
    # Core fields (domain-agnostic)
    strategy_pattern: str
    steps: List[str]
    segments: List[str] = Field(default_factory=list)
    success_rate: float = 1.0
    usage_count: int = 0
    adaptations: List[Dict] = Field(default_factory=list)
    
    # Segmentation metadata
    scope: str = "global"  
    scope_id: Optional[str] = None
    priority: int = 0
    learned_from_count: int = 1
    
    # Domain-specific metrics stored as flexible dict
    domain_metrics: Dict[str, float] = Field(default_factory=dict)
domain_agent = InvestmentAdvisorAgent()

This structure demonstrates the power of domain-agnostic design while maintaining LangMem’s data collection capabilities. Let’s break the code down:

  • Core pattern tracking (universal across domains):
    • strategy_pattern: Identifies the type of situation this procedure addresses – completely domain-neutral (e.g., “moderate risk portfolio rebalancing” for investments, “password reset troubleshooting” for customer service, “visual learning approach” for education)
    • steps: The concrete sequence of actions that have proven successful, expressed as simple strings that any domain can populate
    • segments: Generic tags indicating which user groups benefit from this procedure – not tied to any specific domain terminology
  • Performance metrics (domain-independent):
    • success_rate: Tracks correlation between using this procedure and positive outcomes – universal metric
    • usage_count: Enables temporal pattern detection to see if effectiveness changes over time
    • domain_metrics: A flexible dictionary where each domain stores its specific metrics – investments might track avg_portfolio_performance, education might track comprehension_score, healthcare might track treatment_adherence
  • Hierarchical learning scopes (works for any domain):
    • scope: Indicates the hierarchical level at which this procedure should be applied: “user” for individual personalization, “community” for user segment patterns, “task” for request-type specialization, or “global” for universal best practices – a concept that applies uniformly across all domains
    • scope_id: Identifies the specific user or community when the scope is not global
    • priority: Determines which procedure to use when multiple match (more specific scopes have higher priority)
  • Evolution tracking (critical for all domains):
    • adaptations: Maintains complete version history, enabling rollback if new strategies underperform
    • learned_from_count: Tracks how many conversation examples contributed to this procedure

The benefit of this structure is its flexibility through the domain_metrics dictionary. Each domain can store whatever metrics matter to it without changing the core structure. An investment advisor tracks portfolio performance, a healthcare agent tracks patient outcomes, and an educator tracks learning progress – all using the same DomainProcedure class. In Chapter 19, in the Designing domain metrics: the art of measuring success section, we will discuss how to approach metrics with one goal, many goals, some in between, and the implications for these different strategies.

This structure embodies a critical insight from LangMem: not all patterns should be learned globally, regardless of domain. In our investment advisor example, a retiree needs different strategies than a young professional. In healthcare, morning appointment preferences might be user-specific or common to all elderly patients. In education, visual learning preferences might apply to individual students or entire learning style segments. The structure handles all these cases uniformly.

The adaptations field is particularly important for production safety across all domains. As LangMem notes, this version history enables rollback if new rules degrade performance. Each adaptation entry tracks what changed, when it changed, and what performance metrics justified the change. This creates an audit trail that makes procedural learning transparent and reversible, whether you’re tracking investment returns, patient health outcomes, or student test scores. We will talk more about how to roll things back after the code lab.

Step 3 – initialize hierarchical procedural memory

We’ll create the core learning engine that automates the extraction of successful patterns from conversations. This system demonstrates how LangMem creates a feedback loop where every interaction potentially improves future performance, while maintaining complete separation between the domain-agnostic learning mechanism and domain-specific logic.

domain_agent = InvestmentAdvisorAgent()
investment_memory = ProceduralMemory(
    llm=agent.llm, domain_agent=domain_agent
)

This initialization demonstrates the clean separation of concerns in our architecture:

  • Domain agent encapsulation: InvestmentAdvisorAgent encapsulates all investment-specific logic – prompts, community definitions, success metrics – in the domain_investment directory. This domain agent implements the DomainAgent interface, providing the following:
    • Learning prompts for different scopes (global, user, community, task)
    • Community definitions (conservative retirees, aggressive millennials, moderate professionals)
    • Task identification logic (rebalancing, tax planning, risk assessment)
    • Success scoring calculations specific to investment outcomes
    • Domain-specific metric updates (portfolio performance, rebalancing frequency)
  • Procedural memory system: The ProceduralMemory class is completely domain-agnostic. It receives a domain agent and uses it to do the following:
    • Create learning prompts from the domain’s templates
    • Identify task types using the domain’s categorization
    • Calculate success scores using the domain’s metrics
    • Update domain-specific metrics after performance feedback

This separation means the same ProceduralMemory class works with any domain. To create a healthcare agent, you’d simply pass a HealthcareAgent instance – for education, EducatorAgent. The procedural memory system doesn’t change; only the domain agent implementation differs.

  • Hierarchical storage structure: The system maintains separate storage for each learning scope, with retrieval prioritizing the most specific applicable knowledge (user → community → task → global):
    • user_procedures: Individual user preferences and personalized strategies (highest priority)
    • community_procedures: Patterns for user segments (defined by the domain)
    • task_procedures: Specialized approaches for different query types
    • global_procedures: Universal patterns that work across all users (lowest priority, but always available as a fallback)
  • Learning history tracking: Separate history tracking for each scope enables what LangMem calls “trajectory analysis”:
    • user_learning_history: Records user-specific adaptations
    • community_learning_history: Monitors segment-level learning and tracks discovered segments dynamically as they emerge from data
    • task_learning_history: Captures the evolution of task-specific strategies and their effectiveness across different query types
    • global_learning_history: Tracks all universal pattern discoveries

The initialization confirms our procedural memory system is ready with four hierarchical scopes, but no learned strategies yet. This empty state represents our starting point, which is an agent with the capability to learn but no accumulated knowledge.

  • Domain-agnostic learning process: Notice how the learning process itself is completely domain-agnostic. The learn_from_interaction method does the following:
    • Receives a query and interaction data (universal concepts)
    • Delegates to the domain agent to identify task types and communities
    • Uses domain-provided prompts to extract patterns
    • Stores procedures in the hierarchical structure

The same method would work identically for a healthcare agent learning from patient interactions or an education agent learning from tutoring sessions.

  • Automatic scope determination: The system automatically determines which scopes are appropriate for each interaction:
    • If user_id is provided, it attempts user-level learning
    • If the user belongs to a community, it learns community patterns
    • If the query matches a task type, it develops task-specific strategies
    • Global patterns are always considered for universal insights

This automatic scoping ensures patterns are learned at the most appropriate level without manual intervention.

  • Error resilience: The learning process is designed for production robustness. If learning fails at one scope (perhaps due to LLM parsing errors), it continues learning at other scopes. This resilience ensures that valuable patterns aren’t lost due to isolated failures.

The update_from_performance method in procedural_memory.py handles this feedback processing. Let’s examine its core logic:

def update_from_performance(
    self, strategy: str, performance_data: Dict, scope: str = "global",
    scope_id: Optional[str] = None
) -> Dict:
    """Update procedure based on performance feedback"""
    # Select the appropriate procedure store based on scope
    if scope == "user" and scope_id and scope_id in self.user_procedures:
        procedures = self.user_procedures[scope_id]
    elif scope == "community" and scope_id and scope_id in self.community_procedures:
        procedures = self.community_procedures[scope_id]
    elif scope == "task" and scope_id and scope_id in self.task_procedures:
        procedures = self.task_procedures[scope_id]
    else:
        procedures = self.global_procedures
    for pattern, proc in procedures.items():
        if (
            pattern.lower() in strategy.lower()
            or strategy.lower() in pattern.lower()
        ):
            # Calculate success score using domain-specific metrics
            success_score = self.domain_agent.calculate_success_score(
                performance_data
            )
            old_rate = proc.success_rate
            # Momentum-based update: 80% old rate, 20% new score
            proc.success_rate = min(
                1.0,
                proc.success_rate * 0.8 + success_score * 0.2
            )
            # Let domain agent update its specific metrics
            self.domain_agent.update_domain_metrics(
                proc, performance_data)
            # Record adaptation for audit trail and potential rollback
            proc.adaptations.append({
                "timestamp": datetime.now().isoformat(),
                "performance": performance_data,
                "old_rate": old_rate,
                "new_rate": proc.success_rate,
                "success_score": success_score
            })
            return {
                "updated": pattern,
                "scope": scope,
                "scope_id": scope_id,
                "new_success_rate": round(proc.success_rate, 2),
                "performance_trend": (
                    "improving"
                    if proc.success_rate > old_rate
                    else "declining"
                ),
                "total_adaptations": len(proc.adaptations)
            }
    return {"updated": None}

This method demonstrates several important design decisions. Scope-aware procedure lookup ensures updates are applied at the correct hierarchical level by first determining which procedure store to search based on the provided scope parameter. Rather than requiring exact matches, the method uses case-insensitive substring matching for flexibility in finding relevant procedures even when strategy names vary slightly. The actual success calculation is delegated to the domain agent through calculate_success_score(), keeping the procedural memory system domain-agnostic while allowing each domain to define what “success” means in its context. Finally, the min(1.0, ...) cap ensures success rates never exceed 100%, maintaining valid probability values even after many positive feedback rounds.

This feedback processing demonstrates LangMem’s multi-source feedback triangulation. The domain agent’s calculate_success_score method weighs different success factors according to domain priorities. For investments, returns matter most (50%), followed by satisfaction (30%) and goal achievement (20%). These weights fundamentally shape how the agent evolves. If we changed them to prioritize user satisfaction (50%), returns (30%), and goal achievement (20%), the agent would develop very different strategies. Instead of learning to maximize portfolio performance even if clients don’t fully understand the approach, it would learn to prioritize clear explanations and client comfort, potentially accepting lower returns for higher satisfaction. Over time, this satisfaction-focused agent might develop strategies such as “always explain tax implications in simple terms” or “check for understanding before proceeding,” while a returns-focused agent would learn strategies such as “immediately identify underperforming assets” or “prioritize high-yield opportunities.” The same procedural memory system would produce completely different learned behaviors based on these success metrics, demonstrating how domain-specific values directly influence agent evolution. A healthcare domain might prioritize patient outcomes differently, while education might focus on comprehension scores.

The momentum-based update (80% old, 20% new) prevents single outliers from drastically changing strategies while still allowing adaptation. Notice the following:

  • Good performance improves confidence: Rounds 1-2 with high satisfaction and positive returns increase success rate
  • Poor performance triggers adjustment: Round 3’s poor results immediately lower the success rate
  • Recovery is gradual: Round 4’s moderate success begins recovery, but doesn’t immediately restore high confidence

The adaptation history provides complete auditability – every change is tracked with timestamp, reason, and impact. This enables what LangMem calls “safe rollback” – if a strategy’s performance degrades persistently, we can revert to previous versions.
